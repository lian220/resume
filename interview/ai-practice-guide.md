# AI 활용 면접 대응 연습 가이드

> 목적: 이력서 기반 질문과 처음 보는 신규 문제 모두에서 "AI를 쓰는 사람"이 아니라 "AI를 통제하고 검증하는 엔지니어"로 보이기 위한 훈련 문서.

---

## 1. 전체 포지셔닝

### 핵심 메시지

> 저는 AI를 코드 생성기가 아니라 문제 구조화, 대안 탐색, 테스트 후보 생성, 리뷰 보조 도구로 사용합니다. 최종 판단은 요구사항, 보안, 데이터 일관성, 테스트 결과를 기준으로 제가 책임집니다.

### 면접관이 보고 싶은 것

| 평가축 | 좋은 신호 | 위험한 신호 |
|---|---|---|
| 문제 정의 | 요구사항, 제약, 실패 케이스를 먼저 정리 | 바로 "AI에게 물어보겠습니다" |
| AI 제어력 | AI에게 역할과 출력 형식을 구체적으로 지시 | AI 답변을 그대로 따라감 |
| 검증 습관 | 테스트, 로그, 쿼리, 보안 경계로 확인 | "AI가 맞다고 했습니다" |
| 구현 판단 | 트레이드오프와 제외한 대안을 설명 | 한 가지 방법만 고집 |
| 커뮤니케이션 | 과정과 근거를 면접관에게 계속 공유 | 조용히 구현만 진행 |

## 2. 답변 기본 구조

모든 AI 활용 질문은 아래 6단계로 답한다.

```text
1. 문제를 먼저 재정의한다.
2. AI에 맡길 영역과 직접 판단할 영역을 분리한다.
3. AI에게 구체적 역할과 제약을 준다.
4. AI 결과를 테스트/보안/도메인 규칙으로 검증한다.
5. 틀렸거나 부족한 부분을 보완한다.
6. 최종 의사결정과 남은 리스크를 설명한다.
```

### 30초 답변 템플릿

```text
저는 먼저 요구사항과 실패 케이스를 정리한 뒤, AI에는 대안 비교나 테스트 케이스 후보처럼 탐색 비용이 큰 작업을 맡깁니다.
다만 인증, 결제, 권한처럼 리스크가 큰 영역은 AI가 낸 코드를 그대로 쓰지 않고, 상태 전이·권한 경계·멱등성·트랜잭션 경계를 직접 검증합니다.
예를 들어 추천인 리워드 기능에서는 AI가 일반적인 배치 구조를 제안하더라도, 저는 D+8 환불 제외, 조건부 UPDATE 기반 멱등 지급, Sybil 가능성 같은 도메인 리스크를 기준으로 다시 설계를 조정합니다.
```

## 3. 이력서 기반 질문 대응

### STAR 1: 인증·인가

면접관 의도:
- AI가 만든 보안 설계를 검증할 수 있는가
- JWT, OAuth, RBAC의 trade-off를 설명할 수 있는가
- 권한 상승, stale permission, 헤더 위조 같은 위험을 아는가

좋은 답변 키워드:
- PKCE, RTR, JTI blacklist
- 서버 사이드 이중 검증
- 객체레벨 인가
- Gateway 중앙 RBAC
- PII masking
- Redis TTL + invalidation

예상 질문:

```text
AI가 JWT에 모든 권한을 넣는 설계를 제안했다면 어떻게 판단하겠습니까?
```

답변 방향:

```text
FE UI 분기용 권한과 서버 접근 제어용 권한을 분리하겠습니다.
JWT 권한은 stale 가능성이 있으므로 실제 자원 접근은 DB/Redis 기반 서버 검증으로 다시 확인합니다.
권한 박탈이나 계정 정지는 JTI blacklist 또는 authorization record 폐기와 캐시 무효화로 즉시 차단합니다.
```

### STAR 2: SigNoz 관측성

면접관 의도:
- AI/도구를 도입할 때 비용, 보안, 운영 리스크를 같이 보는가
- 단순 설치가 아니라 팀 표준화까지 했는가

좋은 답변 키워드:
- SaaS vs self-hosted 비용 비교
- 데이터 주권
- OpenTelemetry
- traceId/log correlation
- HA trade-off
- retention policy

예상 질문:

```text
AI가 Datadog 도입을 추천했다면 왜 SigNoz를 선택할 수 있었나요?
```

답변 방향:

```text
도구 추천 자체보다 우리 조직의 제약을 먼저 봤습니다.
비용, 데이터 외부 반출, 팀 규모, 운영 난이도를 비교했고, 서비스 장애와 관측성 도구 장애의 중요도를 분리했습니다.
그 결과 self-hosted SigNoz가 당시 규모에서는 더 합리적이라고 판단했습니다.
```

### STAR 3: ROS 소통관리

면접관 의도:
- 처음부터 도메인을 설계하고 완성할 수 있는가
- 외부 API 실패, 알림톡, 조회 상태, 권한 필터를 다룰 수 있는가

좋은 답변 키워드:
- 3종 컨텐츠 lifecycle
- 본부/점주 양방향 흐름
- 알림톡 retry
- 트랜잭션 경계 분리
- N+1 제거
- 권한 기반 조회 필터

### STAR 4: 멤버십 결제 + 추천인

면접관 의도:
- 돈이 걸린 도메인에서 멱등성, 환불, 어뷰징을 다룰 수 있는가
- AI가 놓치기 쉬운 정책 경계를 잡을 수 있는가

좋은 답변 키워드:
- billing key
- idempotency
- lifecycle state machine
- D+8 reward
- conditional UPDATE
- snapshot
- Sybil risk
- manual retry gate

예상 질문:

```text
AI가 추천인 리워드 배치를 단순히 매일 지급하도록 만들자고 하면 어떻게 보완하겠습니까?
```

답변 방향:

```text
먼저 지급 조건을 결제 안정화 이후로 늦춥니다.
첫 결제 후 D+8과 1시간 마진을 두고, 그 사이 환불이 있으면 지급 대상에서 제외합니다.
또 배치가 중복 실행돼도 조건부 UPDATE로 PENDING 상태를 선점한 경우에만 발급하게 해서 정확히 1회 지급되도록 합니다.
```

## 4. 완전히 새로운 문제 대응

처음 보는 문제에서는 기술을 많이 아는 척보다, 문제를 안전하게 쪼개는 과정이 중요하다.

### 4.1 신규 문제 7단계 프로토콜

```text
1. 목표 확인: 누가, 어떤 상황에서, 어떤 결과를 원하는가
2. 입력/출력 정의: 데이터, API, 화면, 이벤트, 배치 여부
3. 제약 확인: 시간, 보안, 비용, 트래픽, 외부 연동, 개인정보
4. 핵심 도메인 모델 초안: 상태, 주체, 권한, 불변식
5. AI 활용: 대안, 엣지 케이스, 테스트 후보, 누락 리스크 요청
6. 검증: 테스트, 장애 시나리오, 보안, 데이터 정합성 확인
7. 최종 설명: 선택안, 버린 대안, 남은 리스크, 다음 개선
```

### 4.2 면접 중 말로 보여줄 진행 멘트

```text
먼저 문제를 구현 문제로 바로 보지 않고, 도메인 상태와 실패 케이스를 나누겠습니다.
AI에는 제가 정리한 요구사항을 기준으로 엣지 케이스와 테스트 후보를 요청하겠습니다.
다만 최종 설계는 트랜잭션 경계, 권한, 멱등성, 운영 복구 가능성을 기준으로 제가 검증하겠습니다.
```

### 4.3 AI에게 던지는 프롬프트 패턴

#### 요구사항 정리용

```text
너는 시니어 백엔드 리뷰어다.
아래 요구사항에서 누락된 정책, 엣지 케이스, 실패 시나리오를 찾아라.
답변은 "확인 질문 / 위험 / 테스트 케이스 / 설계 대안"으로 나눠라.
```

#### 구현 전 설계 검토용

```text
아래 설계안의 데이터 정합성, 보안, 멱등성, 운영 복구 관점의 약점을 찾아라.
단순한 코드 스타일 지적은 제외하고, 장애나 금전 손실로 이어질 수 있는 리스크를 우선순위로 정렬하라.
```

#### 테스트 후보 생성용

```text
이 기능의 정상 케이스, 경계값, 동시성, 외부 API 실패, 재시도, 권한 실패 테스트 케이스를 표로 만들어라.
각 케이스마다 기대 결과와 확인해야 할 로그/DB 상태를 포함하라.
```

### 4.4 신규 문제에서 반드시 확인할 체크리스트

| 영역 | 확인 질문 |
|---|---|
| 도메인 | 상태 전이가 명확한가? 되돌릴 수 없는 상태가 있는가? |
| 권한 | 누가 누구의 데이터를 볼 수 있는가? 객체레벨 인가가 필요한가? |
| 멱등성 | 같은 요청/배치가 두 번 실행되면 어떻게 되는가? |
| 외부 연동 | 타임아웃, 실패, 부분 성공을 어떻게 복구하는가? |
| 트랜잭션 | DB 변경과 외부 호출의 순서는 안전한가? |
| 보안 | 개인정보, 토큰, 권한 헤더, 입력값 조작 위험은 없는가? |
| 관측성 | 실패했을 때 trace/log/metric으로 원인을 찾을 수 있는가? |
| 운영 | 수동 재처리, 관리자 조회, 알림 채널이 있는가? |

## 5. 신규 문제 연습용 시나리오

### 시나리오 A: AI 기반 고객 문의 자동 분류

문제:

```text
고객 문의를 AI로 자동 분류해 담당 부서에 배정하는 기능을 설계하세요.
오분류, 개인정보, 운영자 수정, 재학습 가능성을 고려하세요.
```

핵심 평가:
- AI 결과를 신뢰하지 않고 검증/수정 흐름을 두는가
- 개인정보 마스킹과 로그 정책을 고려하는가
- 운영자 피드백 루프를 설계하는가

### 시나리오 B: 프로모션 쿠폰 자동 발급

문제:

```text
특정 조건을 만족한 사용자에게 쿠폰을 자동 발급하는 배치를 설계하세요.
중복 발급, 환불, 조건 변경, 배치 재실행을 고려하세요.
```

핵심 평가:
- 멱등키와 상태 전이를 잡는가
- 조건 스냅샷과 live config의 차이를 설명하는가
- 배치 실패 후 재처리를 설계하는가

### 시나리오 C: AI 코드 리뷰 봇 도입

문제:

```text
팀에 AI 코드 리뷰 봇을 도입하려 합니다.
어떤 기준으로 도입하고, 어떤 리뷰는 AI에게 맡기지 않겠습니까?
```

핵심 평가:
- AI에 맡길 일과 사람이 책임질 일을 분리하는가
- 보안, 성능, 도메인 정책 리뷰는 사람 검증이 필요함을 아는가
- 오탐/미탐과 개발자 신뢰 문제를 고려하는가

## 6. 연습 운영 방식

### 1라운드: 이력서 기반

목표:
- 이미 한 일을 AI 시대 언어로 재해석한다.
- "내가 AI보다 잘 판단한 부분"을 보여준다.

진행:

```text
질문 1개
사용자 답변
피드백
개선 답변
다시 답변
```

### 2라운드: 신규 문제

목표:
- 처음 보는 문제에서 당황하지 않고 구조화한다.
- AI 활용 계획과 검증 계획을 동시에 말한다.

진행:

```text
문제 제시
2분 구조화
AI에게 물어볼 프롬프트 말하기
설계 초안 답변
리스크/검증 보완
```

### 3라운드: 압박 질문

목표:
- 약점을 먼저 인정하고 개선안을 말한다.
- 답변이 추상적이면 구체 사례, 수치, 검증 방법으로 끌어내린다.
- "AI를 썼다"가 아니라 "AI가 틀릴 수 있는 지점을 어떻게 통제했는가"를 증명한다.

예시:

```text
AI 없으면 못 푸는 거 아닌가요?
그 설계는 AI가 만든 것과 뭐가 다른가요?
AI가 틀렸는지 어떻게 압니까?
대규모 트래픽 경험은 부족하지 않나요?
```

## 7. Grill Me 모드

Grill Me 모드는 친절한 모의면접이 아니라, 답변의 빈틈을 끝까지 파고드는 압박 훈련이다.

이 모드의 목적은 압박 자체가 아니다. 여러 답변 방식, 문제 접근 방식, AI 활용 방식을 비교한 뒤 **면접에서 가장 설득력 있고 방어 가능한 선택지를 고르는 것**이다.

### 7.1 진행 규칙

```text
1. 질문은 한 번에 하나만 받는다.
2. 답변은 60~90초 안에 한다.
3. 답변 후 바로 꼬리 질문을 받는다.
4. 모르는 것은 모른다고 인정하되, 검증 계획을 말한다.
5. 추상어가 나오면 반드시 실제 사례나 기준으로 바꾼다.
```

### 7.2 비교 검토 방식

하나의 질문에 대해 최소 2개 답변 전략을 비교한다.

```text
전략 A: 짧고 실무적인 답변
전략 B: 이력서 STAR와 연결한 깊은 답변
전략 C: 신규 문제 대응 프로토콜 중심 답변
```

비교 기준:

| 기준 | 확인 질문 |
|---|---|
| 설득력 | 면접관이 "실제로 해본 사람"이라고 느끼는가? |
| 방어력 | 꼬리 질문이 들어와도 무너지지 않는가? |
| 구체성 | 실제 사례, 정책, 검증 기준이 포함되는가? |
| 범용성 | 이력서 기반 질문뿐 아니라 신규 문제에도 적용 가능한가? |
| 리스크 인식 | 약점과 남은 리스크를 인정하는가? |
| 시간 적합성 | 60~90초 안에 말할 수 있는가? |

선택 원칙:

```text
1. 1차 답변은 짧고 구조적으로 말한다.
2. 꼬리 질문이 오면 STAR 사례로 깊이를 보여준다.
3. 신규 문제면 이력서 사례를 억지로 끼우지 말고, 7단계 프로토콜로 접근한다.
4. 보안/결제/권한 질문은 항상 검증 기준과 실패 케이스를 포함한다.
```

### 7.3 답변 전략 비교 예시

질문:

```text
AI를 잘 활용한다는 게 정확히 뭡니까?
```

#### 전략 A: 짧은 원칙형

```text
AI를 잘 활용한다는 건 프롬프트를 잘 쓰는 것보다, AI에 맡길 일과 사람이 책임질 일을 구분하는 것이라고 봅니다.
저는 AI를 대안 탐색과 테스트 후보 생성에 쓰고, 최종 판단은 요구사항, 보안, 데이터 정합성, 운영 리스크 기준으로 직접 합니다.
```

장점:
- 짧고 안전하다.
- 어떤 직무 면접에서도 통한다.

단점:
- 실제 경험의 깊이가 약하게 들릴 수 있다.

#### 전략 B: STAR 연결형

```text
AI를 잘 활용한다는 건 AI가 낸 초안을 검증 가능한 설계로 바꾸는 능력이라고 봅니다.
예를 들어 추천인 리워드 배치를 설계한다면 AI는 일반적인 배치 구조를 빠르게 제안할 수 있습니다.
하지만 저는 돈이 걸린 도메인에서는 D+8 환불 제외, PENDING 상태 조건부 UPDATE 기반 멱등 지급, 실패 시 수동 재처리 같은 기준을 직접 확인합니다.
그래서 AI는 탐색 속도를 높이는 도구이고, 최종 설계 책임은 제가 가져갑니다.
```

장점:
- 이력서의 강점과 바로 연결된다.
- "실제로 해본 사람"처럼 들린다.

단점:
- 신규 문제에서 너무 특정 사례에 갇힐 수 있다.

#### 전략 C: 신규 문제 대응형

```text
처음 보는 문제라면 AI에게 바로 정답을 묻지 않습니다.
먼저 사용자, 상태, 권한, 실패 케이스, 외부 연동 여부를 정리하고, 그 다음 AI에게 누락된 정책과 테스트 후보를 요청합니다.
AI 답변은 멱등성, 트랜잭션 경계, 보안, 운영 복구 가능성 기준으로 걸러냅니다.
```

장점:
- 처음 보는 과제형 면접에 강하다.
- 문제 해결 능력을 보여준다.

단점:
- 본인 경험의 임팩트가 덜 드러난다.

추천:

```text
기본 답변은 전략 A + B 조합이 가장 좋다.
먼저 원칙을 짧게 말하고, 바로 STAR 4나 STAR 1 사례를 붙인다.
면접관이 "새로운 문제라면요?"라고 물으면 전략 C로 전환한다.
```

### 7.4 답변 채점 기준

| 평가 항목 | 통과 기준 | 탈락 신호 |
|---|---|---|
| 자기 주도성 | 내가 문제를 정의하고 AI 사용 범위를 정함 | AI가 제안한 대로 했다고 들림 |
| 검증 가능성 | 테스트, 로그, DB 상태, 정책 불변식으로 확인 | "확인했습니다"만 있고 방법이 없음 |
| 도메인 이해 | 돈, 권한, 개인정보, 운영 리스크를 구분 | 기술 구현만 말함 |
| 약점 인식 | 남은 리스크와 개선안을 같이 말함 | 약점이 없다고 말함 |
| 커뮤니케이션 | 면접관이 따라올 수 있게 순서대로 설명 | 키워드만 나열 |

### 7.5 Grill 질문 사다리

면접관은 아래 순서로 압박을 올린다.

```text
Level 1. 사용 경험 확인
Level 2. 실제 사례 요구
Level 3. AI가 틀린 경우 추궁
Level 4. 보안/결제/운영 리스크 추궁
Level 5. 대안 비교와 포기한 선택지 추궁
Level 6. 본인이 한 일과 AI가 한 일의 경계 추궁
```

### 7.6 대표 Grill 흐름 예시

#### 주제: AI 활용 방식

```text
Q1. 개발 과정에서 AI를 어떻게 활용합니까?
Q2. 그건 그냥 검색 잘하는 것과 뭐가 다릅니까?
Q3. AI가 틀린 답을 준 적은 없습니까?
Q4. 틀렸다는 걸 어떻게 알았습니까?
Q5. 그 판단을 본인이 했다는 증거는 무엇입니까?
Q6. AI 없이도 같은 문제를 풀 수 있습니까?
```

추천 답변 방향:

```text
AI는 탐색 속도를 높이는 도구로 쓰되, 최종 판단 기준은 제가 세웁니다.
예를 들어 추천인 리워드 배치에서는 AI가 일반적인 매일 지급 배치를 제안할 수 있지만, 저는 환불 가능 기간과 중복 지급 리스크를 먼저 봤습니다.
그래서 D+8 이후 지급, 환불 이력 제외, PENDING 상태 조건부 UPDATE 선점으로 멱등성을 보장했습니다.
AI가 놓치기 쉬운 부분은 "돈이 한 번 더 나갈 수 있는가", "배치가 두 번 돌면 어떻게 되는가", "운영자가 복구할 수 있는가" 같은 기준으로 검증합니다.
```

#### 주제: 신규 문제 대응

```text
Q1. 처음 보는 요구사항을 받으면 AI에게 바로 물어봅니까?
Q2. 요구사항을 잘못 이해한 채 AI에게 물으면 어떻게 됩니까?
Q3. 그래서 첫 5분 동안 무엇을 합니까?
Q4. AI에게 어떤 프롬프트를 던집니까?
Q5. AI 답변에서 무엇을 버립니까?
Q6. 최종 설계가 안전하다는 걸 어떻게 설명합니까?
```

추천 답변 방향:

```text
처음 5분은 구현이 아니라 문제 경계를 잡는 데 씁니다.
사용자, 상태, 권한, 실패 케이스, 외부 연동 여부를 먼저 정리하고, 그 다음 AI에게 누락 정책과 테스트 후보를 요청합니다.
AI 답변 중 도메인 정책과 맞지 않거나 트랜잭션 경계가 불명확한 제안은 버립니다.
최종 설계는 상태 전이, 멱등성, 권한, 운영 복구 가능성 기준으로 설명합니다.
```

### 7.7 절대 피해야 할 답변

```text
AI가 빠르게 만들어줘서 생산성이 좋아졌습니다.
```

왜 위험한가:
- 본인의 판단 기준이 보이지 않는다.
- AI 의존적으로 들린다.
- 면접관이 바로 "그럼 AI 없으면 못 하나요?"라고 물을 수 있다.

대체 답변:

```text
AI 덕분에 탐색과 초안 작성 속도는 빨라졌지만, 생산성 향상은 검증 기준을 함께 세웠을 때만 의미가 있다고 봅니다.
저는 AI 결과를 테스트, 도메인 정책, 운영 리스크 기준으로 걸러낸 뒤 적용합니다.
```

## 9. 금지 표현과 대체 표현

| 피할 표현 | 대체 표현 |
|---|---|
| AI가 알아서 만들어줬습니다 | AI가 초안을 만들었고, 저는 정책/보안/테스트 기준으로 검증했습니다 |
| 프롬프트를 잘 쓰면 됩니다 | 문제를 구조화하고 제약을 명확히 주는 것이 먼저입니다 |
| 일단 구현해보고 고칩니다 | 실패 케이스와 데이터 정합성을 먼저 정의하고 구현합니다 |
| AI가 맞다고 해서 썼습니다 | 테스트와 운영 로그, 도메인 규칙으로 맞는지 확인했습니다 |
| 저는 AI를 많이 씁니다 | 저는 AI를 탐색과 검토에 쓰고, 최종 판단은 제가 책임집니다 |
