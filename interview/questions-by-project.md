> 정리일: 2026-06 · 짝 문서: [`questions-by-tech.md`](questions-by-tech.md)(기술 개념별), 이 문서는 **프로젝트(경험)별**
>
> **2026 면접 핵심**: ① 기술+행동 통합 평가(소통=핵심 기술 신호) ② 시니어는 **시스템 설계 + 트레이드오프** ③ "smart 증명"이 아니라 **"믿고 맡겨도 안전한 사람" 증명** ④ AI 활용은 가점·감점 없음 ⑤ 답변 프레임 **SOAR**(Situation-Obstacle-Action-Result).

## 1. 인증·인가 시스템 (SNS 로그인 + RBAC)

**Q1. 외부 IdP(Auth0·Cognito·Firebase) 대신 자체 JWT를 발급한 이유는? 운영 부담을 떠안는 건데.**
- *답변*: 토큰 수명·강제 무효화(계정 정지·권한 박탈)를 직접 통제해야 했고, 두 프로덕트(앱+BO)에 동일 원칙 적용이 필요. 외부 IdP는 SDK 결합·커스텀 상태머신 제약. 단, 키 관리·RTR 구현 비용은 감수 → JTI 블랙리스트·JWKS 캐싱으로 상쇄.

**Q2. 게이트웨이에서 권한을 중앙 집행하면 그게 SPOF 아닌가? 게이트웨이 다운되면 전체 인가가 멈추는데.**
- *답변*: 게이트웨이 다중화 + auth `/rbac/permissions` 호출에 서킷브레이커 + Redis 캐시(SHA-256 토큰 해시 키)로 가용성 확보. 트레이드오프: 다운스트림 JWT 재파싱 제거(6서비스)로 얻는 단순성 vs 관문 집중 리스크 → 관문을 견고화하는 쪽 선택. `[확인: 게이트웨이 다중화 구성]`

**Q3. JWT에 권한 플래그를 담으면 stale 문제가 있다. 권한 회수가 즉시 반영 안 되는데 어떻게?**
- *답변*: JWT 권한은 **FE 분기 전용**, 실제 자원 접근은 서버사이드 DB/캐시 이중검증(OWASP API3). TTL 1h로 stale window 제한 + 강제 무효화 시 JTI 블랙리스트(Redis TTL)로 즉시 차단.

**Q4. RBAC를 3단(ROOT/FUNCTION/ENDPOINT)으로 나눈 이유? 과설계 아닌가?**
- *답변*: 역할 수×권한 수 폭증 방지. 와일드카드 계층 매칭(`HQ.*`)으로 권한 추가 시 매핑 폭증 없이 계층 부여. 객체레벨 인가로 SUPERVISOR가 SUPER_ADMIN 계정 수정 차단(수직 권한 상승).
- *예상 후속*: "수평 권한(다른 브랜드 침범)은 어떻게?" → brandId/storeId 스코프 일치 검증.

## 2. 멤버십 자동결제(토스 빌링) + 추천인

**Q1. 중복 결제를 분산락이 아니라 DB 멱등성으로 막은 이유는?**
- *답변*: 일배치 특성상 동시성 경쟁이 낮고, 분산락(Redis/ZK)은 운영·장애 포인트 추가. `(memberId, billingDate)` 유니크 제약 + 조건부 UPDATE로 **외부 락 없이 정확히 1회**. 비용 대비 합리적.
- *예상 후속*: "락이 정말 필요 없나? 동시 배치 인스턴스가 뜨면?" → 유니크 제약이 최후 방어선이라 중복 INSERT 자체가 실패.

**Q2. 결제 실패 재시도는 자동인데 리워드 지급은 왜 수동 게이트로 뒀나? 일관성 없어 보이는데.**
- *답변*: 결제 재시도는 멱등(중복결제는 유니크로 차단)이라 자동 안전. 리워드는 **돈 지급**이라 blind auto-retry가 이중 지급 위험 → 실패 격리(FAILED) + Slack 알림 + 운영자 확인 후 멱등 수동 재처리. "위험의 비대칭"에 따라 다르게 설계.

**Q3. PCI DSS는 어떻게 대응했나?**
- *답변*: 카드 정보는 시스템에 일절 미저장, 토스 빌링키만 보관(vault 위임)으로 **카드데이터 스코프 자체를 제거**. 카드 변경 시 빌링키 재발급.

**Q4. D+8 리워드 지급 정책의 근거는?**
- *답변*: 첫 결제 + 8일(전액 환불 가능기간 종료) + 1h 안전마진. 환불 존재 시 지급 제외(첫 결제 `MIN(paidAt)` 기준)로 "환불했는데 리워드 받는" 어뷰징 차단.
- *예상 후속*: "지급 후 환불(클로백)은?" → 30일 클로백은 백로그로 인지(솔직히 답변).

## 3. 본부↔점주 소통관리 도메인 (외식UP ROS)

**Q1. 3종 컨텐츠를 STI(단일 테이블 상속)로 모델링한 이유? JOINED나 테이블 분리가 정석 아닌가?**
- *답변*: 3종이 공통 라이프사이클(임시저장→발행→조회→확인)을 공유하고 타입별 필드 발산이 작아 **폴리모픽 목록 조회 최적화**가 핵심. 타입별 nullable은 수용. 발산이 컸다면 JOINED를 택했을 것.
- *예상 후속*: "nullable 컬럼 늘어나면?" → 타입별 검증을 애플리케이션/도메인에서 보강.

**Q2. 알림톡 발송에 Transactional Outbox를 쓴 이유? 그냥 재시도로는 안 됐나?**
- *답변*: 게시글 커밋 + 외부 발송요청은 **dual-write** — 단순 retry는 DB 커밋/발송 사이 실패 시 유실. Outbox로 발송요청을 같은 트랜잭션에 기록 후 분리 처리해 최종 일관성 보장.
- *예상 후속*: "Outbox 폴링 부하는? CDC는 고려했나?" → 규모 기준 폴링으로 충분, 트래픽 증가 시 CDC 전환 여지.

**Q3. 출시 후 성능 튜닝에서 N+1을 어떻게 발견·해결했나?**
- *답변*: 목록/상세/팝업 실행계획 분석 → Fetch Join + Batch Size + 복합 인덱스. 정렬·검색을 DB로 위임해 애플리케이션 필터링 제거. `[확인: P95 N→Mms]`

## 4. 관측성(SigNoz) 자발 도입

**Q1. SigNoz self-hosted 운영 부담을 떠안았는데, Datadog 같은 SaaS가 더 합리적이지 않았나?**
- *답변*: SaaS·AWS 네이티브(X-Ray+CloudWatch) 모두 인제스트 **사용량 비례 변동비** → 트래픽 증가 시 비용 예측 어려움. 기존 운영 EC2 유휴 리소스에 얹어 **증분 비용 ≈ EBS 디스크뿐**, 선형 증가 안 하는 구조라 판단. + 사용자/매출 데이터 외부 반출 회피(데이터 주권).

**Q2. AWS Application Signals로도 trace-log 상관이 되는데 굳이 SigNoz?**
- *답변*: OTel-native라 **백엔드 이식 자유 → X-Ray 벤더락인 회피**가 차별점. 향후 Grafana Tempo/Jaeger 등으로 교체 가능한 구조.

**Q3. 관측성 도구를 단일 인스턴스로 둔 건 위험하지 않나?**
- *답변*: "관측성 다운 ≠ 서비스 다운" — 관측 도구 장애가 서비스 가용성에 직접 영향 없으므로 HA는 과투자로 판단. 단일 EC2 + EBS 스냅샷 백업 + ClickHouse TTL로 충분.
- *예상 후속*: "도구 다운 중 장애가 겹치면?" → 그 시간 로그는 보존(MDC), 사후 분석 가능 + 복구 우선순위 낮음.

**Q4. 전수 계측이 아니라 핵심 흐름만 커스텀 span을 단 이유는?**
- *답변*: span 폭증 → ClickHouse 저장·조회 비용 + 노이즈. OTel Java Agent 자동계측(HTTP/JPA/외부호출) + 비즈니스 핵심(OAuth 콜백, 결제, 리워드 배치)만 선별.

## 5. 머스트잇 — 카탈로그/클러스터링·생성형 AI (직전 경력, 질문 多) ⭐

> 면접관은 **직전 경력을 가장 깊게** 판다. 답변 포인트는 *방향만* 제시했으니, 본인 실제 구현·수치로 **반드시 채워 준비**할 것(사실 기반).

### 5-1. 카탈로그 클러스터링 (품번 기준 상품 매칭)
**Q. 품번 기준 매칭에서 오매칭(동일 품번·다른 상품, 표기 흔들림)은 어떻게 처리했나?**
- *준비 포인트*: 정규화 규칙·예외 케이스·신뢰도 임계. 매칭 정확도를 어떻게 측정했는지(샘플 검수/지표).

**Q. 다량 처리에 메시지큐를 도입한 이유와 선택(Kafka?)? 동기 처리 대비 트레이드오프는?**
- *준비 포인트*: 처리량/백프레셔/순서보장·중복 처리, 큐 도입으로 얻은 것 vs 운영 복잡도.

### 5-2. 생성형 AI 자동화 (ChatGPT 시리얼넘버 추출 → 클러스터링 40%→60%)
**Q. 정규식·룰 대신 LLM을 쓴 이유? 환각/오추출 검증은? (2026 핵심 질문)**
- *준비 포인트*: 비정형 상품명이라 룰로 한계 → LLM. 출력 검증(포맷 체크·매칭 후 대조), 실패 시 폴백. 비용·rate limit 관리.

**Q. 클러스터링 비율 40%→60%는 어떻게 측정했고, 남은 40%의 원인은?**
- *준비 포인트*: 측정 기준(전체 대비 매칭된 비율), 미매칭 원인 분류, 추가 개선 방향.

### 5-3. 외부 크롤링 성능 8H → 1H (동기→비동기)
**Q. 8시간이 1시간으로 — 병목이 정확히 무엇이었고, 비동기화는 어떤 방식(스레드풀/CompletableFuture/WebClient/코루틴)?**
- *준비 포인트*: I/O 대기 병목 → 동시성. Cursor 페이징 + Aggregation을 쓴 이유(메모리·조회 효율). 1H의 측정 방법.

**Q. Redis로 외부 크롤링 횟수를 카운팅 — Rate Limit 회피 로직과 분산 환경 정확성은?**
- *준비 포인트*: 원자적 INCR + TTL, 다중 워커에서 공유 카운터, 차단 시 백오프.

### 5-4. CDC + MSK 커넥터 (상품 등록/수정 실시간 감지)
**Q. 변경 감지를 polling이 아니라 CDC로 간 이유? CDC 솔루션(Debezium 등)·MSK 커넥터 선택 근거는?**
- *준비 포인트*: 실시간성·DB 부하·정합성. 이벤트 순서보장·중복(at-least-once) 처리, 컨슈머 멱등성.

### 5-5. PDP 성능 개선 (이미지 로딩 캐싱, 2초 → 1초 이하)
**Q. 캐싱 위치(CDN/Redis/브라우저)와 무효화 전략은? "기존 로직 유지·최소 공수"의 의미는?**
- *준비 포인트*: 인지 성능 vs 실제 성능, 캐시 키·TTL·무효화, 리스크 최소화한 점진적 적용.

### 5-6. 찜하기 저장 구조 개선 (RDB → Redis + Spring Batch write-back)
**Q. 찜하기를 Redis로 옮긴 이유와, Redis 휘발성에 대한 영속성 보장(write-back 주기·유실 위험)은?**
- *준비 포인트*: RDB 쓰기 부하 → Redis. write-back 배치 주기 동안의 유실 윈도우·복구, AOF/스냅샷 고려 여부.
- *예상 후속*: "Redis 다운 시 찜 데이터는?" → 백업 데이터 저장 전략으로 답.

## 6. 공통 — 행동/시니어십/AI (2026 가중 ↑)

**Q. (SOAR) 가장 어려웠던 의사결정과 그때의 갈등 상황은?**
- *포인트*: "강점만 나열" 금지 — 트레이드오프를 감수한 결정 1개(예: 게이트웨이 중앙집행의 SPOF 리스크 vs 단순성)를 Obstacle 중심으로.

**Q. 1인/소수로 결제·관측성을 책임졌는데, 어떻게 안정성을 담보했나?**
- *포인트*: fail-fast 설계, 실패 격리+수동 게이트, 회귀 0건, 운영 규칙화 — "믿고 맡겨도 안전한 사람" 신호.

**Q. AI 도구를 개발에 어떻게 쓰나? (가점 질문, 감점 없음)**
- *포인트*: 솔직하게. 개인 프로젝트 Alpha Foundry에서 LLM/Vertex AI 통합·1인 풀스택 운영 경험 연결. "AI 잘 쓰려면 결국 개발을 잘해야 한다"는 관점.

**Q. (시스템 설계) 이 결제/추천 시스템을 10배 트래픽으로 확장한다면?**
- *포인트*: 일배치 → 분산 처리(파티셔닝), 멱등성 유지, 빌링 PG rate limit, 리워드 배치 병렬화. SQL vs NoSQL·일관성 vs 가용성 트레이드오프 명시.
