# STAR 4 — 멤버십 추천인코드 시스템 면접 노트

> 출처: `cancun-api` 실제 코드(`feature/CHABYULHWA-4086`) + 백엔드 아키텍트·보안 전문가 리뷰 (2026-06)
> 저자 확인: 추천인 핵심(`ReferralDecisionManager`·`ReferralCodeResolver`·`Reward`·`Batch`·`GrantService`) = 본인(lian) 코드 ✅

---

## 1. 30초 요약 (이 시스템이 뭐냐)
멤버십 구독 위에 얹은 **친구 추천 그로스 루프**. 추천코드를 입력하면 ① 가입자는 **할인**, ② 추천인은 첫 결제가 안정화된 **D+8에 리워드(포인트/쿠폰)**. 어뷰징 방지를 위해 쿨다운·자기추천 차단·환불 제외·멱등 지급을 코드로 강제.

## 2. 외워야 할 숫자·정책 (틀리면 안 됨)
| 항목 | 값 | 이유 |
|---|---|---|
| 추천 트랙 쿨다운 | **999개월 (생애 1회)** | 추천 할인 어뷰징 방지 (정책 3-4) |
| 일반 캠페인 쿨다운 | **12개월** | 첫달0원 등 1년 1회 (정책 3-3) |
| 쿨다운 키 | **(사업자번호, 종사업자번호, plan)** | 종사업자번호는 `coalesce('')`로 NULL-safe |
| 리워드 지급 시점 | **첫 결제 + 8일(D+8) + 1h 마진** | 전액환불 가능기간 종료 후 → 환불 전 지급 방지 |
| 환불 처리 | **D+8 이내 환불 = 지급 제외 / 지급 후 환수 없음** | 30일 클로백은 백로그(미구현) |
| 할인 하한 | **0원 클램프** (1원 아님) | PG가 1원 결제를 거부 → 1원이 아니라 0원 |
| 리워드 상태 | **PENDING → GRANTED** (2개뿐) | GRANTED는 영구 skip(환수 X) |
| 리워드 PK | **(membershipId, promotionId)** | billing 차원 없음 = 1딜당 1리워드 |

## 3. 아키텍처 한 줄 흐름
`ReferralCodeResolver`(코드 해석·스냅샷) → `ReferralDecisionManager`(적용/보류 결정·쿨다운·자기추천 차단) → 가입 시 `MembershipReferralReward`(PENDING) 생성 → **D+8 배치**(`BatchProcessor`)가 대상 선별 → `GrantService`가 **조건부 UPDATE로 선점 후 같은 트랜잭션에서 발급**.

## 4. 면접에서 먼저 꺼낼 4대 강점 설계 (왜 좋은지까지)
1. **멱등 "선점 후 발급"** — `UPDATE ... SET GRANTED WHERE status=PENDING`의 **row count로 선점** → 같은 트랜잭션에서 포인트/쿠폰 발급. **외부 분산락 없이 정확히 1회**. (동시 배치 2개가 돌아도 진 쪽은 row 0 → skip)
2. **가입 시점 리워드 스냅샷** — 가입~D+8 사이 BO가 설정을 바꿔도, **약속한 시점의 리워드**를 지급. 배치는 저장된 데이터만으로 동작(live 조인 불필요).
3. **체험→유료 자동전환 1회용 슬롯** — 체험은 아직 결제가 없으니 reward를 만들 수 없음. 코드를 `membership_recommender_username` 슬롯에 보관 → 자동전환 첫 결제 때 적용·기록·슬롯 비움. **reward는 항상 실결제 위에서만 생성**되는 불변식 유지.
4. **실패 격리 설계** — 지급 실패 = `PENDING→FAILED` **종료**(자동 무한재시도 X) + **Slack 능동 알림** + BO 조회 가시화 → 운영자 원인 조치 후 **멱등 수동 재처리**. 돈이 걸린 리워드라 blind auto-retry보다 "수동 게이트"가 안전 — 의도된 결정.

## 5. 내가 먼저 인정할 약점 2개 (선제적으로 말하면 가점)
> "이 시스템 트랜잭션 코어는 탄탄한데, **운영·경계 쪽에 알고 있는 갭이 있습니다**" 라고 먼저 꺼내라.
>
> ⚠️ **프레이밍 주의 (캡)**: "한 추천인이 여러 명에게 추천받아 리워드 여러 번" 은 **버그가 아니라 어필리에이트 의도**(파워 추천인 허용). 쿠폰 per-user limit 무시도 "추천 성공마다 쿠폰"이라 의도된 것. **추천인당 캡은 그로스 모델을 훼손하므로 정답이 아님.**
>
> ✅ **실패 처리는 이미 해결됨** (= 강점, 약점 아님): 지급 실패 = `recordFailure`로 **`PENDING→FAILED` 종료**(자동 재시도 X) + **Slack 에러채널 알림** + BO 조회 API 가시화 → 운영자가 원인 조치 후 **멱등 수동 재처리**. 돈이 걸린 리워드라 blind auto-retry보다 안전한 결정. *(코드 일부 stale 주석 "PENDING 유지로 재시도"는 옛 버전 잔재 — 실제는 FAILED 종료)*

| # | 약점 | 한 줄 설명 | 개선안 |
|---|---|---|---|
| 1 | **가짜 피추천인 자가파밍(Sybil)** | 정상 다수추천은 OK(의도). 문제는 *같은 운영자*가 다른 사업자번호 shell 가입자를 만들어 본인 코드 자가소비 → D+8 후 환불. 자기추천은 `userId`만 차단해 shell은 통과 | **캡(X) — 그로스 훼손.** fraud 군집 휴리스틱(카드BIN·기기·IP·사업자번호 군집) + 클로백 |
| 2 | **`MIN(paidAt)` 취약** | 같은날 재시도 tie / 환불후 재청구 시 첫 결제 오인 가능, projection `distinct()` 없음 | 첫 결제 식별을 생성순/플래그로, tie·재청구 테스트 |

> **클로백(환수)은 "버그" 아니라 "범위 결정"으로 답하라**: D+8 후 환불 시 리워드 유지 = 기획 3-4/5-5에서 **D+8·무환수로 확정**한 현 스코프. 30일 클로백(GTM OPR 6-4/D7)은 **다채널 확장 시 백로그**. 단일 USER_REFERRAL 채널은 D+8(전액환불기간 종료)로 충분히 방어.
> 핵심: 실제 잔여 위험은 "**가짜 피추천인 + 환불타이밍**" 조합. 장기적으로 fraud 휴리스틱 + 클로백.

## 6. 면접 예상질문 Top 12 (모범답변 요지)
1. **멱등 어떻게 보장?** → 조건부 UPDATE(PENDING→GRANTED) row lock으로 선점, 발급을 같은 트랜잭션에. 분산락·outbox 불필요. "정확히 1회"는 *모두 한 트랜잭션에서 커밋되기 때문*.
2. **쿠폰은 per-user limit 무시인데 중복발급 안 나나?** → 유일 방어가 그 선점 1개. in-tx라 happy path는 정확히 1회. 발급이 트랜잭션 밖(async/다른 datasource)으로 나가는 순간 깨짐 → reward PK 멱등키 추가가 defense-in-depth.
3. **지급 실패하면?** → `grant` 롤백(미지급) → `recordFailure`로 **`PENDING→FAILED` 종료**(자동 재시도 X) + Slack 에러채널 알림 + BO 조회. 운영자가 원인 조치 후 **멱등 수동 재처리**(`markGranted`가 PENDING만 전이라 중복 안전). 돈이라 blind auto-retry 대신 수동 게이트가 의도.
4. **D+8 지나고 환불하면?** → 추천인 리워드 유지(환수 없음). D+8은 *자동 전액환불 기간*만 방어, 고의 churn은 못 막음. 30일 클로백이 정답(백로그).
5. **왜 D+8?** → 첫 결제 전액환불 가능기간 종료 + 1h 마진. 환불 확정 전 지급 방지.
6. **왜 스냅샷? live config 안 읽고?** → 약속 시점 보장 + 배치가 순수함수. 단 쿠폰은 metaNumber(포인터)만 스냅샷이라 내용은 live → POINT는 완전 freeze, COUPON은 반-freeze (정직하게 인정).
7. **쿨다운 키가 가입자 기준인데 추천인 어뷰징은?** → 정상 다수추천은 의도(어필리에이트). 진짜 리스크는 *가짜 피추천인 자가파밍*(다른 사업자번호 shell). 자기추천은 userId만 차단 → **fraud 휴리스틱 + 클로백**으로 대응(블랭킷 캡 아님 — 정상 추천인 훼손).
8. **종사업자번호 NULL은?** → 입력 `orEmpty()` + 쿼리 `coalesce('')` 양쪽 정규화 → NULL-vs-빈문자 우회 없음. (사업자번호는 정규화 안 해서 형식차이 우회 여지 → 개선점)
9. **체험→유료 슬롯, 결제 실패하면 슬롯은?** → 슬롯 비움이 같은 트랜잭션이면 롤백돼 보존(다음 시도 가능). 단 명시적 테스트 없음 → 추가 필요.
10. **리워드 PK에 billing 차원이 없는데?** → 1딜당 1리워드 보장(멱등키 안정). 대신 회차별/반복 리워드는 스키마 변경 필요. 현재 "첫 결제·생애1회" 정책엔 최소·정확.
11. **배치 동시 실행/재실행은?** → 2중 방어: row level(선점) + run level(executionId 중복 거부 + job lock). 새 executionId 재실행은 PENDING 재처리(=의도된 재시도, 선점 덕에 안전).
12. **Calculator에서 추천 할인은 어떻게 적용?** → 추천/코드형 프로모션은 *가격 계산기에서 제외*(`isCodeGatedPromotion` 필터)하고 `decide()`→record→`buildPlanWithPromotion`으로 강제 적용. 일반 프로모션만 자동 최저가 계산.

## 7. 절대 틀리면 안 되는 정정 (예전 표현 → 실제)
- ❌ "정액 할인 **1원** 보정" → ✅ **0원 클램프** (PG가 1원을 거부해서 1원이 아니라 0원)
- ❌ "환불 시 리워드 **회수**" → ✅ **지급 전 환불이면 배치에서 제외 / 지급 후 환수는 없음**(클로백 백로그)
- ❌ "idempotency = **(memberId,rewardDate) DB 유니크 제약**" → ✅ **조건부 UPDATE(PENDING→GRANTED) 선점**, PK는 (membershipId, promotionId)
