STAR 1

사용자 인증·인가 시스템 풀스택 설계 (SNS 로그인 + RBAC)

⭐⭐ 메인
모바일 앱 소셜 로그인(Kakao · Apple OAuth + PKCE)부터 B2B 백오피스 RBAC까지 — 두 프로덕트의 인증·인가를 동일 원칙으로 설계·배포한 풀사이클 경험
Situation
  • 차별화상회는 기존 ID/PW 가입만 있었고, 카카오·애플 소셜 로그인을 도입하면서 준회원/정회원/탈퇴/연동 계정 합류 같은 복잡한 상태 흐름이 필요했음.
  • 외식UP ROS 백오피스는 본부/점주/관리자 권한이 코드 곳곳에 흩어져 있어 Role-Permission-User 매핑 기반 RBAC가 필요.
  • 두 서비스의 인증·인가를 한 사람이 설계하면서 일관된 의사결정(자체 JWT, 명시적 권한 모델, 캐시 무효화 책임)을 적용.
Task
  • 사용자(앱) 측: OAuth 2.0 + PKCE → 자체 JWT 발급, 역할/상태 기반 인가, 계정 합류 흐름.
  • 운영자(BO) 측: Role-Permission-User 매핑 모델, 권한 캐시 + invalidation, BO 화면 페어.
Action ① 사용자 인증
  • OAuth 2.0 + PKCE → 자체 JWT: 카카오·애플 각각 Authorization Code + PKCE(S256) 플로우로 토큰 교환 (Implicit Grant 배제, OAuth 2.1 방향), Spring Authorization Server 파이프라인으로 자체 JWT(1h) + Refresh Token(14d) 발급
  • Apple client_secret 자체 생성: ES256 서명 JWT를 BE에서 직접 생성 (6개월 만료 제한 — 만료 N일 전 자동 재생성 스케줄 적용), JWKS 공개키 캐싱 + 검증 실패 시 즉시 재조회 fallback(cache-miss-on-validation-failure)
  • Refresh Token Rotation(RTR): 갱신 시 1회용으로 발급하고, 구 refresh 토큰 재사용 시 토큰 탈취로 간주해 세션 전체(Refresh Token Family) 무효화 — replay attack 방어
  • OAuth 콜백·Refresh 엔드포인트 Rate Limiting 적용으로 brute-force 방어
Action ② 사용자 인가 + 계정 상태 머신
  • 역할 기반 JWT: ROLE_TEMP_USER, SOCIAL_USER, USER, MEMBER 등 역할을 JWT payload에 포함, Spring Security @PreAuthorize로 자원 보호
  • 계정 상태 머신: SOCIAL_CREATED → SOCIAL_PROFILE_COMPLETED → BASE_USER_LINKED → MEMBER_CONVERTED 4단계 전이를 enum으로 명시적 모델링, 단계마다 접근 가능한 API 차등화
  • 권한 플래그 토큰 동봉 (의식적 trade-off): canViewWholesalePrice, canOrder 같은 비즈니스 권한을 JWT 응답에 동봉해 FE 분기 단순화. 단, JWT 페이로드의 권한은 FE UI 분기 전용이며 실제 자원 접근은 서버 사이드에서 DB/캐시 기반 이중 검증 (OWASP API3 대응). JWT TTL 1h로 stale window 제한, 강제 무효화는 JTI 블랙리스트(Redis TTL)로 즉시 차단
  • 계정 합류: 충돌 시 HTTP 409 + 합류 후보 응답 스펙 설계 (민감 정보 최소 노출 + CSRF 토큰 검증), 합류 동의 시 단일 트랜잭션으로 통합
  • 상태 게이팅 API: 부정 전환 방지를 위해 별도 게이팅 엔드포인트 추가
  • 탈퇴 시 provider Deauthorization: 카카오·애플 SDK Deauthorization 호출, 외부 호출 실패 시 비동기 재시도 큐로 최종 일관성 확보
  • 공격 표면 축소(Attack Surface Reduction): 초기 분리했던 API 8건을 검토 단계에서 기존 API에 흡수 통합 — 노출 엔드포인트 수 감소로 보안·운영 복잡도 동시 절감
Action ③ 운영자 RBAC (외식UP ROS 본부 백오피스, UR10)
  • 3단 권한 모델 (Role - Permission - User): 권한을 ROOT(시스템 루트) / FUNCTION(평가식, 예 HQ.SUPER_ADMIN.UPDATE_ACCOUNT) / ENDPOINT(GET:/api/... 패턴) 3종으로 타입화. 역할 6종(SFN_ROOT · SUPER_ADMIN · BRAND_MANAGER · SUPERVISOR · STORE_OWNER · STORE_STAFF)에 role_priority 계층 부여, ros_role_permission_mapping/ros_user_role_mapping으로 N:M 연결.
  • 와일드카드 계층 권한 매칭: PERMISSION_*(전체) · HQ.*(본부 전체) · HQ.SUPER_ADMIN.READ(단건) 접두 와일드카드를 지원하는 matchesPermission() — 권한 폭증 없이 계층적 부여.
  • 함수레벨 + 객체레벨 이중 인가 (OWASP API1/API3 · IDOR 방어): @PreAuthorize("hasPermission(#rosUserId, 'TYPE_ROS_USER', 'UPDATE_ACCOUNT')") 한 줄로 ① 호출자의 기능 권한 + ② 수정 대상 유저의 역할까지 조회해 HQ.{대상역할}.UPDATE_ACCOUNT 보유 여부를 객체 단위로 재검증 → SUPERVISOR가 SUPER_ADMIN 계정을 수정·삭제·잠금해제 못 하도록 수직 권한 상승을 코드로 차단 (CustomPermissionEvaluator + 도메인별 DomainPermissionEvaluator 체인).
  • 응답 PII 마스킹 AOP: @MaskIfNoPermission(permission="ROOT.PHONE_NUMBER_READ", maskType=PHONE_MIDDLE_PARTIAL) + 컨트롤러 응답을 가로채는 DtoMaskingAspect — 권한 없는 운영자에겐 전화번호를 010-****-5678로 자동 마스킹. 인가를 진입점뿐 아니라 응답 직렬화 시점까지 다층 적용.
  • API 게이트웨이 중앙 RBAC 집행 + 헤더 기반 전파: Spring Cloud Gateway(RbacAuthorizationFilter)를 단일 인가 관문으로 설계 — ① Auth 서버 /rbac/permissions로 토큰→권한 1회 해석(Redis 캐시, SHA-256 토큰 해시 키·TTL 10분) ② ENDPOINT_{METHOD}:{PATH} 권한을 AntPathMatcher로 중앙 엔드포인트 인가(+화이트리스트로 공개 엔드포인트 허용) ③ 검증된 권한을 X-Hq-Roles · X-User-Roles · X-Function-Permissions · X-Endpoint-Permissions · X-Ros-Permissions 헤더로 주입 + X-Gateway-Header 신뢰 마커 부착 → 다운스트림 6개 서비스(auth·shop·hq·notes·admin)는 JWT 재파싱 없이 게이트웨이가 검증한 헤더만 신뢰. 클라이언트가 보낸 권한 헤더는 게이트웨이가 권위 값으로 덮어써 위조 차단, 다운스트림은 X-Gateway-Header 유무로 경유 판별(미경유 시 자체 RbacHeaderInjectingFilter fallback). 기존 JWT 직접 파싱 컨버터는 @Deprecated 전환해 권한 해석 책임을 게이트웨이로 단일화.
  • Redis 분산 권한 캐시 + 명시적 무효화 API: 역할→권한 캐시(permissionByRole, TTL 1분) + 토큰→RBAC 응답 캐시(SHA-256 토큰 해시 키, TTL 10분)를 Redis로 운영 — 멀티 인스턴스에서도 단일 캐시 공유로 노드별 stale 원천 차단. 역할 부여 시 @CacheEvict 자동 무효화 + BO에서 권한 수정 직후 호출하는 전체 무효화 API → 최대 1분 TTL 대기 없이 즉시 반영.
  • 권한 매핑 배치 + 감사 추적: 대량 매핑 추가/삭제 배치 엔드포인트, 권한 삭제 시 ros_permission_deleted에 JSONB 스냅샷 백업으로 audit trail 보존.
  • 강제 무효화 (BO측 경로): 계정 정지·권한 박탈 시 oauth2_authorization 레코드 폐기 + Redis 권한 캐시 무효화로 즉시 차단 — 앱측 JTI 블랙리스트와 구분된 BO 전용 경로.
Result

  • 정회원 전환율 출시 전 대비 N% 개선.
  • QA 이슈 약 N건 출시 후 안정화, 이후 인증 관련 운영 장애 0건.
  • API 8건 통합 결정으로 노출 엔드포인트 수 감소 — 공격 표면 축소 + 신규 입사자 인증 도메인 온보딩 시간 단축.
  • ROS RBAC 19/22건 완료, 권한 변경 후 BO 반영: 캐시 TTL 자연만료 시 최대 1분 → 무효화 API 호출 시 즉시(0초) 전 인스턴스 반영.
  • 두 서비스에 "역할 + 명시적 권한 모델 + 캐시·세션 무효화 책임" 동일 원칙 적용 — 인증·인가 일관성 확보.

STAR 2

SigNoz 관측성 시스템 도입 — 자발 제안 → 팀 표준 정착

⭐
운영 장애 디버깅 병목을 스스로 발견 → OpenTelemetry 기반 self-hosted SigNoz 제안·도입 → 팀/전사 관측성 표준 정착 (SaaS 대비 연 N만원 절감)
Situation
  • 운영 환경에서 장애·지연 발생 시 로그 grep으로만 원인 추적. 분산 API 호출 / 외부 연동(카카오·애플·FreshON) / 배치 흐름을 한눈에 볼 수 없어 디버깅에 평균 N시간 이상 소요.
  • Datadog · New Relic 도입 검토 — 비용 부담(예상 연 N만원)과 사용자/매출 데이터 외부 반출 우려로 보류 상태.
Task
  • 팀 차원의 관측성 표준 부재를 본인이 문제로 제기하고, 비용·데이터 주권 모두 만족하는 대안을 직접 선정 → 도입 → 확산까지 책임.
Action
  • 문제 제기 & 제안: 운영 장애 디버깅 사례를 근거로 팀에 공유, OpenTelemetry 기반 self-hosted APM 도입 제안.
  • 기술 선정 (SaaS vs OSS): Datadog 과금 모델(호스트 + 인덱싱 볼륨) · New Relic(데이터 인제스트 GB) 비용 산정 → SigNoz self-hosted(EC2 인스턴스 사양 연 N만원) 비교. Trace/Log/Metric 단일 UI 통합 조회 + OTel 네이티브 + 벤더 종속 회피 기준으로 SigNoz 채택 설득.
  • 인프라 구축: EC2 단독 + docker-compose로 배포. 팀 규모·트래픽 기준에서 단일 인스턴스로 충분했고, 관측성 도구 다운 ≠ 서비스 다운이므로 HA 비용 대비 합리적 선택. 데이터 백업은 EBS 스냅샷 일 1회 자동화, 외부 노출은 SG IP 제한 / VPN / 프록시로 차단. ClickHouse 보존 정책: 트레이스 N일 / 메트릭 N일 / 로그 N일, TTL 정책으로 디스크 자동 관리.
  • 계측: OTel Java Agent가 GlobalOpenTelemetry를 초기화하고 HTTP / JPA / 외부 호출 자동 계측. 핵심 비즈니스 흐름(OAuth 콜백, 결제 Calculator, 리워드 배치 등)에는 @WithSpan / tracer.spanBuilder()로 커스텀 span 선별 추가. parent context 전파 확인 완료.
  • 운영화: API별 에러율·P95 응답시간 대시보드 구성, 알람 임계값: P95 > Nms / 에러율 > N% 기준 Slack 알림. traceId/spanId를 Logback MDC에 주입해 로그 → Trace 연관 조회 가능.
  • 팀 확산 & 표준화: 도입 가이드 문서 N페이지 작성, 온보딩 세션 N회 진행. N명 팀원 자발 활용, N개 신규 서비스에 OTel Agent 적용이 디폴트 절차로 편입.
Result
  • 팀/전사 관측성 표준 정착: N명 팀원 자발 확산, N개 신규 서비스 디폴트 적용.
  • 운영 장애 평균 디버깅 시간 N시간 → M분 단축 — 분산 트레이스로 외부 호출 지연·N+1 쿼리·배치 병목 즉시 식별.
  • SaaS APM 대비 연 N만원 절감.
  • 데이터 주권 확보: 사용자/매출 trace가 모두 사내 인프라에 머무름.
  • OTel 표준 계측으로 벤더 종속 없는 관측성 스택 — 향후 Grafana Tempo / Jaeger 등 다른 backend나 OTel Profiling(4th signal) 확장 가능한 구조.
KEYWORDS
OpenTelemetry Observability SigNoz (self-hosted) Distributed Tracing EC2 + docker-compose ClickHouse TTL 보존 정책 OTel Java Agent + @WithSpan traceId MDC 로그 연관 벤더락인 회피 / 데이터 주권 EBS 스냅샷 백업 팀 표준 정착 Performance Tuning
STAR 3

외식UP ROS 소통관리 — 본부 ↔ 점주 양방향 도메인 신규 구축

⭐
ROS 1.0 신규 출시에 맞춰 본부 ↔ 점주 양방향 소통관리 (3종 컨텐츠 + 1:1 문의 + 알림톡) 도메인을 제로에서 N개월 만에 BE 단독 풀스택 구축 → 출시 후 성능 튜닝까지 단독 완수
Situation
  • ROS 1.0 신규 출시에서 본부와 점주를 잇는 커뮤니케이션 채널이 부재. 공지 미전달, 동의서 누락, 공문 확인 불가 등 운영 투명성 리스크 존재.
  • 단순 공지 하나가 아니라 공지(Notice) / 동의서(Agreement) / 공문(Document) 3종 컨텐츠 타입, 각각의 라이프사이클(임시저장 → 발행 → 조회 → 동의/확인완료), 카카오 알림톡 발송 + 1:1 문의 thread까지 묶인 도메인이 필요.
Task
  • BE 단독으로 본부향 + 점주향 + 관리자 Admin 3축, 3종 컨텐츠, 알림톡 / 1:1 문의 / 조회현황 전 영역의 API 설계·구현. 후속 운영 성능 이슈도 동일 책임.
Action ① 기반 설계 & 본부향 풀 CRUD
  • 도메인 설계: 3종 컨텐츠 + 1:1 문의 통합 모델 + 라이프사이클 정의
  • 본부향 CRUD: 목록 / 상세 / 작성 / 수정 / 삭제 / 임시저장 / 공개·비공개 토글
  • 공문 확인 → 읽음 처리 변경: 사용자 행동 분석 반영해 도메인 의미 재정의
Action ② 점주향 양방향 흐름
  • 점주향 API: 목록 / 상세 / 동의서 동의완료·동의취소 / 공문 확인완료 / 조회완료
  • 1:1 문의 thread: 점주 등록·수정·삭제, 본부 응답·삭제, 본부향 목록에 미답변 개수 노출
Action ③ 알림톡(Kakao AlimTalk) 연동
  • 발송 API + 템플릿 조회 + 인사말 + 알림 현황·조회 현황 + 버튼 링크 기능
  • 발송 실패 시 재시도: 외부 API 호출 실패 시 exponential backoff 재시도 + 상태 추적, DB 트랜잭션과 외부 호출의 경계를 분리해 부분 실패 시에도 게시글 상태는 일관성 유지
Action ④ 출시 후 운영 단계 성능 개선
  • 쿼리 성능 튜닝: 목록 / 상세 / 팝업 리스트 실행계획 분석 후 N+1 제거(쿼리 N개 → M개 감소) + 복합 인덱스 추가(대상 컬럼) + Fetch Join/Batch Size 조정
  • 정렬·검색 옵션을 DB로 위임해 애플리케이션 필터링 제거
  • 권한 캐시 즉시 무효화 (STAR 1의 RBAC와 연계)
Result
  • 제로에서 N개월 만에 39건 API 풀스택 BE 단독 구축 — 본부향 26 + 점주향 13, 전건 배포완료.
  • 본부-점주 간 공지 미전달·동의서 누락 리스크 해소 — 운영 투명성 확보.
  • QA 이슈 14건 전건 종결.
  • 성능 개선: 목록 P95 N→M ms (N% 단축).
  • 본부 N명 / 점주 N명, 알림톡 월 N건 발송 규모로 안정 운영.

STAR 4

차별화멤버십 자동결제(빌링) 시스템 + 추천인코드 그로스 루프

⭐
토스페이먼츠 빌링키 기반 정기결제 시스템을 BE 단독 설계·구현 — 멤버십 라이프사이클 상태머신, 실패 재시도, 안분 환불, 무료체험, ERP 회계 연동, 추천인코드 그로스 루프까지 결제 도메인 풀스택
Situation
  • 차별화멤버십(B2B 식자재 정기 구독)을 신규 출시하면서 카드 1회 등록 → 매월 자동결제 흐름과, 가입·연체·해지·재가입을 다루는 멤버십 라이프사이클 관리가 필요.
  • 결제 실패·환불·무료체험·추천인 할인 등 결제 도메인 전체를 BE 단독으로 책임. 사내 ERP(FreshON)와의 회계 동기화까지 묶임.
Task
  • PG(토스페이먼츠) 빌링 API 연동 → 빌링키 발급·관리 → 정기결제 일배치 → 실패 재시도·연체 상태 전환 → 환불·부분환불·무료체험 → 회계 ERP 동기화 → 추천인코드 그로스 루프까지 결제 도메인 풀 사이클 책임.
Action ① 토스페이먼츠 빌링키 발급·관리
  • 빌링키 발급 플로우 설계: FE가 토스 SDK로 카드 정보 입력 → 발급된 authKey를 BE로 전달 → BE가 토스 빌링 API로 빌링키 교환 후 DB에 저장
  • PCI DSS 인지: 카드 정보 자체는 BE 시스템에 일절 저장 안 함 — 토스 빌링키만 저장 (vault 위임). 카드 변경 시 빌링키 재발급
  • 빌링키 무결성: 결제 시점에 빌링키 유효성 사전 검증, 만료·실효 시 사용자에게 카드 재등록 알림
Action ② 정기결제 일배치 + 실패 재시도
  • 결제 일배치 (Spring Scheduler): 매일 정각에 당일 결제 대상 멤버십 조회 → 토스 빌링 API 호출 → 결과 기록. idempotency 키로 (memberId, billingDate) DB 유니크 제약을 걸어 배치 재실행 시 중복 결제 차단
  • 결제 실패 재시도 정책: 실패 시 N일 간격(예: 1일 → 3일 → 7일) exponential backoff 재시도, N회 실패 후 연체 상태로 자동 전환
  • 트랜잭션 경계 분리: 토스 API 호출과 DB 상태 변경 트랜잭션을 분리, 외부 응답 지연·타임아웃 시에도 DB 일관성 유지 + 재처리 가능한 구조
Action ③ 멤버십 라이프사이클 상태머신
  • 상태 정의: 가입 대기 → 활성(active) → 연체(overdue) → 해지 예약(cancelled_scheduled) → 해지(cancelled) → 재가입 으로 enum 기반 상태 머신 명시적 모델링
  • 월 말 해지 정책: 즉시 해지가 아닌 다음 결제일 직전까지 사용 가능 + 자동 해지 처리 (사용자가 이미 결제한 기간은 보장)
  • 연체 → 자동 해지 전환: 재시도 N회 모두 실패 시 자동 해지, 그 시점에 누적 멤버십 권한(STAR 1의 canViewWholesalePrice·canOrder) 즉시 회수
Action ④ 환불·부분환불·무료체험
  • 안분(일할) 환불: 중도 해지 시 사용 일수 기반 일할 계산 — 미사용 잔여 일수만큼 부분 환불
  • 무료체험(Free Trial): 첫 N일 0원 + 체험 종료 시 자동 정기결제 전환. 체험 기간 중 해지 시 결제 미발생
  • 환불 후처리: 환불 발생 시 결제·환불 history 마킹 → 추천인 리워드(아래 ⑥) 회수 배치가 마킹 기반으로 자동 처리
Action ⑤ ERP 회계 동기화 (FreshON 연동)
  • 결제·환불 이벤트 발생 시 사내 ERP(FreshON)로 회원 ERP 코드 등록·수정 동기화 — 외부 호출 실패 시 비동기 재시도 큐로 최종 일관성 확보
  • 멤버십 매출 조회 + 엑셀 다운로드 Admin API로 회계팀 운영 자동화
Action ⑥ 추천인코드 그로스 루프 (멤버십 결제 위에 얹은 마케팅 레버)
  • 결제 Calculator 분기 확장: 정률/정액 할인 분기 추가 — 정책 종류가 2개·확장 가능성 낮아 Strategy 패턴 over-engineering 회피, 기존 Calculator 테스트 커버리지 부재로 리팩터링 리스크 > 분기 비용
  • 정액 할인 1원 보정 로직: PG 최소 결제금액 이슈(0원 결제 거부) 회피
  • D+8 리워드 지급 일배치: 결제 확정일(=결제일+8) 기준, (memberId, rewardDate) DB 유니크 제약으로 idempotency 보장. 환불 시 history 마킹 기반으로 회수
  • 추천코드 관리 Admin API: 코드 발급·활성/비활성 토글, 리워드 지급 내역 조회
Result
💳 차별화멤버십 자동결제 시스템 BE 단독 출시 — 카드 1회 등록만으로 매월 자동 결제, 운영 개입 없이 안정 운영. 결제·환불·리워드 회수 흐름 회귀 0건.
  • 결제 일배치 매일 자동 실행, 카드 등록 회원의 결제 성공률 N%, 실패 시 N일 간격 재시도로 운영 공수 제거.
  • 멤버십 라이프사이클 자동 관리 — 본인 해지·연체·재가입 등 운영자 수동 개입 거의 없음.
  • 추천코드 경유 신규 가입 비율 N%, D+8 리워드 지급 자동화로 마케팅 운영 공수 제거.
  • 결제 도메인 안정성 = 매출 안정성으로 직접 연결.
KEYWORDS
토스페이먼츠 빌링 API 빌링키(Billing Key) 발급·관리 PCI DSS 인지 (카드 정보 vault 위임) 정기결제 일배치 (Spring Scheduler) 결제 실패 재시도 (exponential backoff) 멤버십 라이프사이클 상태머신 (enum) 월 말 해지 예약 안분(일할) 환불 무료체험 (Free Trial) 결제 Calculator 분기 D+N 리워드 배치 idempotency (DB 유니크 제약) 환불 history 마킹 ERP 회계 동기화 (FreshON) Payment Domain Subscription Billing

1인가를 왜 게이트웨이에서 중앙 집행?서비스마다 토큰 검증·권한 해석 중복 제거. Auth 서버가 토큰 검증 권위, 게이트웨이가 권한 1회 해석+캐시+엔드포인트 인가 → 다운스트림은 헤더만 신뢰. 권한 모델 변경 시 게이트웨이 한 곳만 수정
1다운스트림이 헤더만 믿으면 클라이언트가 위조하면?게이트웨이가 토큰 검증 후 모든 권한 헤더를 권위 값으로 덮어씀(클라 입력 무시) + X-Gateway-Header 마커 부착. 다운스트림은 마커로 게이트웨이 경유 확인 (게이트웨이 우회 직접 호출은 네트워크/SG 레벨 차단 전제)
1계정 합류 409 응답에 민감 정보 노출은?합류 후보에 매장명·마스킹된 ID만 포함, 이메일·전화번호 비노출
1RBAC 캐시 — 수평 확장 시 다른 노드는?Redis 분산 캐시로 모든 인스턴스가 단일 캐시 공유. invalidation API 호출 시 즉시 반영 (브로드캐스트 불필요)
1왜 in-memory(Caffeine) 대신 Redis를 택했나?멀티 인스턴스에서 in-memory는 노드별 stale 발생 → invalidation 전파 복잡도. Redis는 단일 진실 원천으로 단순·안전
1사용자측 인증과 운영자측 RBAC를 왜 통합 안 했나?도메인·권한 모델 복잡도 다름, 공통 원칙만 적용이 오버엔지니어링 회피
1대규모 트래픽 경험은?서비스 규모에 맞는 실용적 아키텍처 선택, 단일 서버 + 수직 확장으로 운영 안정성 확보
2왜 Datadog 안 썼나?비용(호스트+인덱싱 볼륨 과금) + 데이터 외부 반출 + OTel 표준 종속성 회피
2EC2 단독이면 SigNoz 다운 시 어떻게?SPOF 인지. 관측성 다운 ≠ 서비스 다운, EBS 스냅샷 백업, CloudWatch 기본 알람 병행
2OTel Collector는 어떻게 구성?SigNoz docker-compose 기본 포함 Collector 활용, 샘플링/라우팅은 [실제 구성]
2Agent + SDK 동시에 쓰면 문제는?GlobalOpenTelemetry 중복 초기화 주의, @WithSpan 남발 시 span 폭증 — 핵심 흐름만 선별
2eBPF 기반 도구와 비교하면?JVM 환경에서 OTel Agent가 애플리케이션 레이어 컨텍스트(JPA 쿼리, 비즈니스 span) 더 풍부
3N+1 어떻게 진단?SigNoz 트레이스에서 DB span 수 확인 + JPA show_sql + 실행계획
3알림톡 발송 실패 시 재시도는?exponential backoff + 상태 추적, DB 트랜잭션과 외부 호출 경계 분리
4토스페이먼츠 빌링키 발급 흐름은?FE 토스 SDK로 카드 입력 → authKey 수령 → BE가 토스 빌링 API로 빌링키 교환 → DB에 빌링키만 저장
4카드 정보를 BE에 저장하나?저장 안 함. PCI DSS 인지하고 vault(토스)에 위임, BE는 빌링키만 보유
4결제 실패 시 재시도 정책은?exponential backoff (1일 → 3일 → 7일 등), N회 실패 시 연체 상태 자동 전환
4멤버십 상태머신을 enum vs 별도 테이블?enum 기반 — 전이가 유한·단순, 상태별 가드 로직이 코드 레벨에 집중 (계정 상태머신과 동일 원칙)
4즉시 해지 vs 월 말 해지?다음 결제일 직전까지 사용 가능 + 자동 해지 — 사용자가 이미 결제한 기간 보장
4중도 해지 시 환불은?사용 일수 기반 안분(일할) 계산 + 미사용 잔여 일수만큼 부분 환불
4무료체험 종료 시 자동 결제는?체험 종료일에 자동 정기결제 전환. 체험 중 해지 시 결제 미발생
4결제 일배치 중복 실행 방어는?(memberId, billingDate) DB 유니크 제약 — 배치 재실행 시 중복 결제 차단
4토스 API 호출과 DB 트랜잭션 경계는?외부 호출과 DB 트랜잭션 분리 — 응답 지연·타임아웃 시에도 DB 일관성 유지 + 재처리 가능
4ERP(FreshON) 동기화 실패 시?비동기 재시도 큐로 최종 일관성 확보 — 결제 자체는 성공 처리
4Calculator 분기를 왜 Strategy 패턴 안 썼나?정책 2종(정률/정액)뿐이고 확장 가능성 낮음, 기존 테스트 커버리지 부재로 리팩터링 리스크 > 분기 비용
4D+8 배치 실패 시 어떻게?idempotency 키 = (memberId, rewardDate) DB 유니크 제약, 재실행 시 중복 skip
4환불 시 리워드 회수는 동일 배치? 별도?동일 배치 내 회수 대상 마킹 후 별도 회수 처리
