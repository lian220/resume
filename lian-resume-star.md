## STAR 1. 사용자 인증·인가 시스템 풀스택 설계 (SNS 로그인 + RBAC) ⭐⭐ — **메인**

> **한 줄 헤드라인**
> 모바일 앱 소셜 로그인(Kakao · Apple OAuth + PKCE)부터 B2B 백오피스 RBAC까지 — 두 프로덕트의 **인증·인가를 동일 원칙으로 설계·배포**한 풀사이클 경험

**Situation**
- 차별화상회는 기존 ID/PW 가입만 있었고, 카카오·애플 소셜 로그인을 도입하면서 **준회원/정회원/탈퇴/연동 계정 합류** 같은 복잡한 상태 흐름이 필요했음.
- 외식UP ROS 백오피스는 본부/점주/관리자 권한이 코드 곳곳에 흩어져 있어 **Role-Permission-User 매핑** 기반 RBAC가 필요.
- 두 서비스의 인증·인가를 한 사람이 설계하면서 **일관된 의사결정** (자체 JWT, 명시적 권한 모델, 캐시 무효화 책임)을 적용.

**Task**
- **사용자(앱) 측**: OAuth 2.0 + PKCE → 자체 JWT 발급, 역할/상태 기반 인가, 계정 합류 흐름.
- **운영자(BO) 측**: Role-Permission-User 매핑 모델, 권한 캐시 + invalidation, BO 화면 페어.

**Action — ① 사용자 인증**
- **OAuth 2.0 + PKCE → 자체 JWT**: 카카오·애플 각각 **Authorization Code + PKCE(S256)** 플로우로 토큰 교환 (Implicit Grant 배제, OAuth 2.1 방향), **Spring Authorization Server** 파이프라인으로 자체 JWT(1h) + Refresh Token(14d) 발급
- **Apple `client_secret` 자체 생성**: ES256 서명 JWT를 BE에서 직접 생성 (6개월 만료 제한 — 만료 N일 전 자동 재생성 스케줄 적용), **JWKS 공개키 캐싱** + 검증 실패 시 즉시 재조회 fallback(cache-miss-on-validation-failure)
- **Refresh Token Rotation(RTR)**: 갱신 시 1회용으로 발급하고, 구 refresh 토큰 재사용 시 토큰 탈취로 간주해 **세션 전체(Refresh Token Family) 무효화** — replay attack 방어
- **OAuth 콜백·Refresh 엔드포인트 Rate Limiting** 적용으로 brute-force 방어

**Action — ② 사용자 인가 + 계정 상태 머신**
- **역할 기반 JWT**: `ROLE_TEMP_USER`, `SOCIAL_USER`, `USER`, `MEMBER` 등 역할을 JWT payload에 포함, Spring Security `@PreAuthorize`로 자원 보호
- **계정 상태 머신**: `SOCIAL_CREATED` → `SOCIAL_PROFILE_COMPLETED` → `BASE_USER_LINKED` → `MEMBER_CONVERTED` 4단계 전이를 enum으로 명시적 모델링, 단계마다 접근 가능한 API 차등화
- **권한 플래그 토큰 동봉 (의식적 trade-off)**: `canViewWholesalePrice`(도매가 조회), `canOrder`(주문) 같은 비즈니스 권한을 JWT 응답에 동봉해 FE 분기 단순화. **단, JWT 페이로드의 권한은 FE UI 분기 전용이며 실제 자원 접근은 서버 사이드에서 DB/캐시 기반 이중 검증** (OWASP API3 대응). JWT TTL 1h로 stale window를 제한하고, 강제 무효화(계정 정지·권한 박탈) 시 **JTI 블랙리스트(Redis TTL)로 즉시 차단**
- **계정 합류**: 충돌 시 **HTTP 409 + 합류 후보 응답 스펙** 설계 (민감 정보 최소 노출 + CSRF 토큰 검증), 합류 동의 시 단일 트랜잭션으로 통합
- **상태 게이팅 API**: 부정 전환 방지를 위해 별도 게이팅 엔드포인트 추가
- **탈퇴 시 provider Deauthorization**: 자체 탈퇴 시 카카오·애플 SDK Deauthorization 호출로 좀비 매핑 방지, 외부 호출 실패 시 비동기 재시도 큐로 최종 일관성 확보
- **공격 표면 축소(Attack Surface Reduction)**: 초기 분리했던 API 8건을 검토 단계에서 기존 API에 흡수 통합 — 노출 엔드포인트 수 감소로 보안·운영 복잡도 동시 절감

**Action — ③ 운영자 RBAC (UR10)**
- **Role - Permission - User 매핑 모델 설계** + Admin BO 화면 페어 구현
- **권한 매핑 배치 API**: 대량 매핑 추가/삭제를 배치 엔드포인트로 분리 — 단건 반복 호출 대비 감사 로그(audit trail) 추적 용이
- **권한 캐시 invalidation API (Redis 기반)**: **Redis 분산 캐시**(TTL [채워주세요: N분])로 권한 조회 성능 확보 — **멀티 인스턴스 환경에서도 단일 캐시 공유로 stale 문제 원천 차단**. 권한 변경 시 즉시 무효화하는 명시적 API 추가 → 운영자가 BO에서 권한 수정 직후 호출하면 모든 인스턴스에 즉시 반영.

**Result**
- **신규 가입자 일평균 10명 → 150명 (15배 / +1,400%)** — 소셜 로그인 도입으로 기존 ID/PW 6단계 가입 흐름을 3단계로 단축, 모바일 가입 마찰의 가장 큰 병목을 제거.
- 정회원 전환율 [채워주세요: 출시 전 대비 N% 개선 또는 절대 수치].
- QA 이슈 약 [채워주세요: N건] 출시 후 안정화, 이후 인증 관련 운영 장애 0건.
- API 8건 통합 결정으로 노출 엔드포인트 수 감소 — 공격 표면 축소 + 신규 입사자 인증 도메인 온보딩 시간 단축.
- ROS RBAC 19/22건 완료, 권한 변경 후 BO 반영 [채워주세요: N분 → 즉시].
- 두 서비스에 **"역할 + 명시적 권한 모델 + 캐시·세션 무효화 책임"** 동일 원칙 적용 — 인증·인가 일관성 확보.

**키워드**: `OAuth 2.0 / OAuth 2.1 (PKCE S256)` · `Apple Sign-In (ES256, JWKS, cache-miss-on-validation-failure)` · `Kakao Login` · `Spring Authorization Server` · `자체 JWT + Refresh Token Rotation (RTR)` · `JTI 블랙리스트 (Redis TTL)` · `역할 기반 JWT (ROLE_*)` · `계정 상태 머신 (enum)` · `Account Linking (409 합류 스펙)` · `Provider Deauthorization` · `RBAC (Role-Permission-User)` · `Redis 분산 권한 캐시 + invalidation API` · `Attack Surface Reduction` · `Rate Limiting` · `OWASP API3 이중 권한 검증` · `Observability` · `System Architecture`

## STAR 3. 외식UP ROS 소통관리 — 본부 ↔ 점주 양방향 도메인 신규 구축 ⭐

> **한 줄 헤드라인**
> ROS 1.0 신규 출시에 맞춰 **본부 ↔ 점주 양방향 소통관리** (3종 컨텐츠 + 1:1 문의 + 알림톡) 도메인을 제로에서 [채워주세요: N개월] 만에 BE 단독 풀스택 구축 → 출시 후 성능 튜닝까지 단독 완수

**Situation**
- ROS 1.0 신규 출시에서 **본부와 점주를 잇는 커뮤니케이션 채널이 부재**. 공지 미전달, 동의서 누락, 공문 확인 불가 등 본부-점주 간 운영 투명성 리스크 존재.
- 단순 공지 하나가 아니라 **공지(Notice) / 동의서(Agreement) / 공문(Document) 3종 컨텐츠 타입**, 각각의 라이프사이클(임시저장 → 발행 → 조회 → 동의/확인완료), **카카오 알림톡 발송** + **1:1 문의 thread**까지 묶인 도메인이 필요.

**Task**
- BE 단독으로 **본부향 + 점주향 + 관리자 Admin 3축**, **3종 컨텐츠**, **알림톡 / 1:1 문의 / 조회현황** 전 영역의 API 설계·구현. 후속 운영 성능 이슈도 동일 책임.

**Action — ① 기반 설계 & 본부향 풀 CRUD**
- **도메인 설계**: 3종 컨텐츠 + 1:1 문의 통합 모델 + 라이프사이클 정의
- **본부향 CRUD**: 목록 / 상세 / 작성 / 수정 / 삭제 / **임시저장** / 공개·비공개 토글
- **공문 확인 → 읽음 처리 변경**: 사용자 행동 분석 반영해 도메인 의미 재정의

**Action — ② 점주향 양방향 흐름**
- **점주향 API**: 목록 / 상세 / **동의서 동의완료·동의취소** / **공문 확인완료** / 조회완료
- **1:1 문의 thread**: 점주 등록·수정·삭제, 본부 응답·삭제, 본부향 목록에 미답변 개수 노출

**Action — ③ 알림톡(Kakao AlimTalk) 연동**
- 발송 API + 템플릿 조회 + 인사말 + 알림 현황·조회 현황 + 버튼 링크 기능
- **발송 실패 시 재시도**: 외부 API 호출 실패 시 exponential backoff 재시도 + 상태 추적, DB 트랜잭션과 외부 호출의 경계를 분리해 **부분 실패 시에도 게시글 상태는 일관성 유지**

**Action — ④ 출시 후 운영 단계 성능 개선 (Performance Tuning)**

## STAR 4. 차별화멤버십 자동결제(빌링) 시스템 + 추천인코드 그로스 루프 ⭐

> **한 줄 헤드라인**
> **토스페이먼츠 빌링키 기반 정기결제 시스템**을 BE 단독 설계·구현 — 멤버십 라이프사이클 상태머신, 실패 재시도, 안분 환불, 무료체험, ERP 회계 연동, 추천인코드 그로스 루프까지 결제 도메인 풀스택

**Situation**
- 차별화멤버십(B2B 식자재 정기 구독)을 신규 출시하면서 **카드 1회 등록 → 매월 자동결제** 흐름과, 가입·연체·해지·재가입을 다루는 **멤버십 라이프사이클 관리**가 필요.
- 결제 실패·환불·무료체험·추천인 할인 등 **결제 도메인 전체**를 BE 단독으로 책임. 사내 ERP(FreshON)와의 회계 동기화까지 묶임.

**Task**
- PG(토스페이먼츠) 빌링 API 연동 → 빌링키 발급·관리 → 정기결제 일배치 → 실패 재시도·연체 상태 전환 → 환불·부분환불·무료체험 → 회계 ERP 동기화 → 추천인코드 그로스 루프까지 **결제 도메인 풀 사이클** 책임.

**Action — ① 토스페이먼츠 빌링키 발급·관리**
- **빌링키 발급 플로우 설계**: FE가 토스 SDK로 카드 정보 입력 → 발급된 `authKey`를 BE로 전달 → BE가 **토스 빌링 API로 빌링키 교환** 후 DB에 저장
- **PCI DSS 인지**: 카드 정보 자체는 **BE 시스템에 일절 저장 안 함** — 토스 빌링키만 저장 (vault 위임). 카드 변경 시 빌링키 재발급
- **빌링키 무결성**: 결제 시점에 빌링키 유효성 사전 검증, 만료·실효 시 사용자에게 카드 재등록 알림

**Action — ② 정기결제 일배치 + 실패 재시도**
- **결제 일배치 (Spring Scheduler)**: 매일 정각에 당일 결제 대상 멤버십 조회 → 토스 빌링 API 호출 → 결과 기록. **idempotency 키**로 `(memberId, billingDate)` DB 유니크 제약을 걸어 배치 재실행 시 중복 결제 차단
- **결제 실패 재시도 정책**: 실패 시 **N일 간격(예: 1일 → 3일 → 7일) exponential backoff 재시도**, **N회 실패 후 연체 상태로 자동 전환**
- **트랜잭션 경계 분리**: 토스 API 호출과 DB 상태 변경 트랜잭션을 분리, 외부 응답 지연·타임아웃 시에도 DB 일관성 유지 + 재처리 가능한 구조

**Action — ③ 멤버십 라이프사이클 상태머신**
- **상태 정의**: `가입 대기` → `활성(active)` → `연체(overdue)` → `해지 예약(cancelled_scheduled)` → `해지(cancelled)` → `재가입` 으로 enum 기반 상태 머신 명시적 모델링
- **월 말 해지 정책**: 즉시 해지가 아닌 **다음 결제일 직전까지 사용 가능** + 자동 해지 처리 (사용자가 이미 결제한 기간은 보장)
- **연체 → 자동 해지 전환**: 재시도 N회 모두 실패 시 자동 해지, 그 시점에 누적 멤버십 권한(STAR 1의 `canViewWholesalePrice`·`canOrder`) 즉시 회수

**Action — ④ 환불·부분환불·무료체험**
- **안분(일할) 환불**: 중도 해지 시 사용 일수 기반 일할 계산 — 미사용 잔여 일수만큼 부분 환불
- **무료체험(Free Trial)**: 첫 N일 0원 + 체험 종료 시 자동 정기결제 전환. 체험 기간 중 해지 시 결제 미발생
- **환불 후처리**: 환불 발생 시 **결제·환불 history 마킹** → 추천인 리워드(아래 ⑥) 회수 배치가 마킹 기반으로 자동 처리

**Action — ⑤ ERP 회계 동기화 (FreshON 연동)**
- 결제·환불 이벤트 발생 시 사내 ERP(FreshON)로 회원 ERP 코드 등록·수정 동기화 — **외부 호출 실패 시 비동기 재시도 큐**로 최종 일관성 확보
- 멤버십 매출 조회 + 엑셀 다운로드 Admin API로 회계팀 운영 자동화

**Action — ⑥ 추천인코드 그로스 루프 (멤버십 결제 위에 얹은 마케팅 레버)**
- **결제 Calculator 분기 확장**: 정률/정액 할인 분기 추가 — 정책 종류가 2개·확장 가능성 낮아 Strategy 패턴 over-engineering 회피, 기존 Calculator 테스트 커버리지 부재로 리팩터링 리스크 > 분기 비용
- **정액 할인 1원 보정 로직**: PG 최소 결제금액 이슈(0원 결제 거부) 회피
- **D+8 리워드 지급 일배치**: 결제 확정일(=결제일+8) 기준, `(memberId, rewardDate)` DB 유니크 제약으로 idempotency 보장. 환불 시 history 마킹 기반으로 회수
- **추천코드 관리 Admin API**: 코드 발급·활성/비활성 토글, 리워드 지급 내역 조회

**Result**
- **차별화멤버십 자동결제 시스템 BE 단독 출시** — 카드 1회 등록만으로 매월 자동 결제, 운영 개입 없이 안정 운영.
- 결제 일배치 매일 자동 실행, **카드 등록 회원의 결제 성공률 [채워주세요: N%]**, 실패 시 N일 간격 재시도로 운영 공수 제거.
- 멤버십 라이프사이클 자동 관리 — 본인 해지·연체·재가입 등 운영자 수동 개입 거의 없음.
- 추천코드 경유 신규 가입 비율 [채워주세요: N%], D+8 리워드 지급 자동화로 마케팅 운영 공수 제거.
- 결제·환불·리워드 회수 흐름 회귀 0건 — **결제 도메인 안정성 = 매출 안정성**으로 직접 연결.

**키워드**: `토스페이먼츠 빌링 API` · `빌링키(Billing Key) 발급·관리` · `PCI DSS 인지 (카드 정보 vault 위임)` · `정기결제 일배치 (Spring Scheduler)` · `결제 실패 재시도 (exponential backoff)` · `멤버십 라이프사이클 상태머신 (enum)` · `월 말 해지 예약` · `안분(일할) 환불` · `무료체험(Free Trial)` · `결제 Calculator 분기` · `D+N 리워드 배치` · `idempotency (DB 유니크 제약)` · `환불 history 마킹` · `ERP 회계 동기화 (FreshON)` · `Payment Domain` · `Subscription Billing`
