> **자기소개 (숫자 후킹)**
> 소셜 로그인으로 신규 가입을 **일 10명 → 150명(15배)**으로 끌어올리고, **결제·환불·리워드 회귀 0건**의 멤버십 빌링 시스템을 BE 단독으로 설계한 **6년 차 백엔드 엔지니어**. 구현 자체보다 **"왜 이 설계를 골랐나(트레이드오프)"**를 기록하고, 관측성·장애 대응으로 **운영 안정성을 끝까지 책임지는 것**을 지향합니다.

**핵심 역량**
- 앱 OAuth + 백오피스 RBAC를 **동일 원칙으로 설계한 인증·인가 풀스택**
- 멱등성·정합성을 갖춘 **결제(빌링) 도메인 단독 설계**
- 팀 관측성 표준 부재를 발견 → **SigNoz 자발 도입·정착** (오너십)
- 신규 도메인 **1인 풀스택 구축 + 출시 후 성능 튜닝**

## 1. 사용자 인증·인가 시스템 풀스택 (SNS 로그인 + RBAC) · 스마트푸드네트웍스

**P — 왜 풀어야 했나**
차별화상회는 ID/PW **6단계 가입**만 있어 모바일 가입 마찰이 가장 큰 이탈 병목이었고, 외식UP ROS 백오피스는 **권한이 코드 곳곳에 흩어져** 본부/점주/관리자 통제가 불가능했다. 두 프로덕트의 인증·인가를 **한 사람이 일관된 원칙으로** 풀어야 했다.

**A — 대안 분석 & 선택 (트레이드오프)**
- 토큰: Implicit Grant 대신 **OAuth 2.1 PKCE(S256) → 자체 JWT** 채택 — 외부 PG/IdP 종속 대신 토큰 수명·무효화를 직접 통제. 구 Refresh 재사용 시 **세션 전체 무효화(RTR)**로 탈취 방어.
- 권한 전파: JWT에 권한을 다 담는 방식(stale 위험) 대신 **API 게이트웨이 중앙 RBAC 집행 + 서버사이드 이중검증** 선택 — JWT 권한은 FE 분기 전용, 실제 접근은 DB/캐시로 재검증(OWASP API3). stale window는 TTL + 무효화 API로 관리.
- 권한 모델: 권한 폭증을 막기 위해 **3단(ROOT/FUNCTION/ENDPOINT) + 와일드카드 계층 매칭**, 객체레벨 인가로 수직 권한 상승 차단.

**R — 결과 (수치)**
신규 가입 **일 10명 → 150명 (15배 / +1,400%)**, 준회원 → 정회원 **전환율 30.2%**(9,723건 중 2,936건), 인증 관련 운영 장애 **0건**, 다운스트림 6서비스 JWT 재파싱 제거.

## 2. 멤버십 자동결제(토스 빌링) + 추천인 그로스 루프 · 스마트푸드네트웍스

**P — 왜 풀어야 했나**
차별화멤버십(B2B 식자재 정기 구독) 출시로 **카드 1회 등록 → 매월 자동결제** + 가입·연체·해지·재가입 라이프사이클이 필요했고, 결제 실패·환불·추천 리워드까지 **결제 도메인 전체를 BE 단독**으로 책임져야 했다. 돈이 걸린 흐름이라 **중복 결제·이중 지급은 곧 사고**.

**A — 대안 분석 & 선택 (트레이드오프)**
- 카드 정보: 자체 보관(PCI DSS 부담) 대신 **토스 빌링키만 저장(vault 위임)** — 규정 리스크 제거.
- 중복 방지: 분산락 대신 **DB 유니크 제약(idempotency 키) + 조건부 UPDATE** — 외부 락 인프라 없이 정확히 1회. 토스 API와 DB 트랜잭션 **경계 분리**로 외부 지연에도 일관성.
- 리워드: blind auto-retry(무한 재시도로 이중 지급 위험) 대신 **실패 격리(FAILED 종료) + Slack 알림 + 수동 멱등 재처리** — "돈은 자동 재시도보다 수동 게이트가 안전".

**R — 결과 (수치)**
멤버십 자동결제 **BE 단독 출시**, 결제·환불·리워드 회수 흐름 **회귀 0건**. *[확인 필요: 카드 등록 회원 결제 성공률 N%, 추천코드 경유 신규 가입 N%]*

## 3. 본부↔점주 양방향 소통관리 도메인 신규 구축 (외식UP ROS) · 스마트푸드네트웍스

**P — 왜 풀어야 했나**
ROS 1.0 출시에서 **본부와 점주를 잇는 채널이 부재** — 공지 미전달·동의서 누락·공문 확인 불가로 운영 투명성 리스크. 단순 공지가 아니라 **공지/동의서/공문 3종 + 알림톡 + 1:1 문의**가 묶인 도메인이 필요했다.

**A — 대안 분석 & 선택 (트레이드오프)**
- 모델링: 타입별 테이블 분리 대신 **STI(단일 테이블 상속)** — 공통 라이프사이클 공유·폴리모픽 목록 조회 최적화(타입별 nullable 수용).
- 알림톡 정합성: 단순 retry로는 dual-write 유실 위험 → **Transactional Outbox**로 게시글 커밋과 발송요청을 한 트랜잭션에 기록 후 분리 처리.
- 출시 후 실행계획 분석 → **N+1 제거 + 복합 인덱스 + Fetch Join**.

**R — 결과 (수치)**
제로에서 **39건 API BE 단독 구축**(본부향 26 + 점주향 13, 전건 배포완료), QA 이슈 **14건 전건 종결**, 공지·동의 누락 리스크 해소. *[확인 필요: 구축 기간 N개월, 목록 P95 N→Mms]*

## 4. 관측성(Observability) 자발 도입 → 팀 표준 정착 · 스마트푸드네트웍스

**P — 왜 풀어야 했나**
운영 장애·지연 시 **로그 grep으로만 원인 추적** — 분산 호출·외부 연동·배치 흐름을 한눈에 못 봐 디버깅 병목. Datadog·New Relic은 **비용 + 사용자/매출 데이터 외부 반출** 우려로 보류 상태였고, **팀 차원 관측성 표준 자체가 부재**했다.

**A — 대안 분석 & 선택 (트레이드오프)**
- SaaS(Datadog 호스트+볼륨 과금 / New Relic GB 과금) vs OSS 비용·데이터 주권 비교 → **SigNoz self-hosted 채택을 직접 설득**(Trace/Log/Metric 단일 UI + OTel 네이티브 + 벤더 종속 회피).
- 가용성: HA 구성 대신 **단일 EC2 + docker-compose** — "관측성 다운 ≠ 서비스 다운"이라 HA 비용은 과투자로 판단.
- 계측: 전수 계측(span 폭증) 대신 **핵심 비즈니스 흐름만 커스텀 span**, traceId를 MDC 주입해 로그↔Trace 연관.

**R — 결과 (수치/정성)**
로그 grep 의존 → **분산 트레이스 단일 조회로 근본원인 추적**, 사용자/매출 trace를 사내 인프라에 유지(**데이터 주권 확보**), 팀 표준으로 정착. *[확인 필요: SaaS 대비 연 N만원 절감, 디버깅 N시간→M분, 확산 N명/N개 서비스]*
