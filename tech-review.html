1. 인증·인가
🔥 Q1-1. Spring Auth Server에서 카카오 로그인 후 자체 JWT 발급까지 어떤 컴포넌트를 커스터마이징했나?
왜: 키워드만 쓴 사람 vs 실제로 다룬 사람 구분.
핵심: AuthenticationSuccessHandler에서 OAuth2User→내부 User 매핑(JIT) → OAuth2TokenCustomizer<JwtEncodingContext>로 role·권한 커스텀 클레임 주입. Authorization Server(토큰 발급)와 oauth2Login()(Resource Owner 인증) 역할 분리 설명하면 가점.
🔥 Q1-2. 운영자 권한 검증은 정확히 어느 레이어에서? (실제 코드)
왜: "DB/캐시 확인한다"만으론 부족. OWASP API1/API3 방어 증명.
핵심(3중 집행): ① 게이트웨이 RbacAuthorizationFilter가 ENDPOINT_{METHOD}:{PATH} 권한을 AntPathMatcher로 중앙 인가 → ② 함수레벨 @PreAuthorize hasPermission(...) CustomPermissionEvaluator → ③ 객체레벨 DomainPermissionEvaluator 대상 객체 역할 재검증. 코드 경로로 답 가능 = 강점.
Q1-3. JWT Access Token을 클라이언트 어디에 저장하나? (localStorage vs httpOnly cookie)
왜: XSS vs CSRF 트레이드오프 = JWT 보안의 절반.
핵심: httpOnly+Secure+SameSite=Lax 쿠키 → XSS 탈취 불가(CSRF는 SameSite로 차단). localStorage → XSS 즉시 탈취. 모바일은 Keychain/Keystore. "우리는 [선택]+[위협모델 근거]".
Q1-4. 강제 무효화(계정 정지)는 어떻게? — 앱측 JTI vs BO측 경로 (실제 코드)
왜: stateless JWT 강제 무효화 한계. ※ 정직성: caribbean-api(BO)엔 JTI 블랙리스트 테이블 없음 — 경로가 다름.
핵심(경로 분리): BO측은 계정 정지·권한 박탈 시 oauth2_authorization 폐기 + Redis 권한캐시 무효화로 즉시 차단. 앱측은 JTI 블랙리스트(Redis TTL, 별도 레포). JTI TTL = AT exp + skew 마진으로 역산. "두 서비스가 무효화 책임을 각자 맞는 방식으로"가 정답.
Q1-5. RTR에서 동시 Refresh 요청(race condition)이 들어오면?
왜: RTR 이론적 약점. 동시 요청이 reuse로 오인돼 세션 전체 무효화.
핵심: Redis SET NX/분산 락 또는 DB SELECT FOR UPDATE로 직렬화. 규모 작으면 "발생 빈도 낮고 재로그인으로 복구 가능한 수용된 위험" 정직하게.
Q1-6. Apple .p8 개인키는 어디에 저장하고 유출되면?
왜: 비대칭키 관리 = PKI 핵심. 유출 시 임의 client_secret 생성으로 우회.
핵심: Parameter Store(SecureString)/Secrets Manager + IAM Role. 유출 시 Apple 포털 즉시 폐기+새 키+배포. 기존 "N일 전 자동 재생성"으로 긴급 교체 동일 파이프라인.
🔥 Q1-8. SUPERVISOR가 SUPER_ADMIN 계정을 수정·삭제하는 수직 권한 상승은 어떻게 막나? (IDOR, 실제 코드)
왜: OWASP API1(BOLA/IDOR). 함수 권한만 보면 "누구의 계정인지"는 놓치는 흔한 구멍.
핵심: @PreAuthorize hasPermission(#rosUserId, 'TYPE_ROS_USER', 'UPDATE_ACCOUNT')가 ① 호출자 기능권한 + ② 대상 유저 역할을 조회해 HQ.{대상역할}.UPDATE_ACCOUNT 보유 여부까지 객체 단위 재검증. role_priority 계층으로 하위가 상위 객체를 못 건드림. 실제 구현이라 자신 있게. ⚠️ 2차: role_priority는 수직만 — 다른 브랜드 침범(수평 IDOR)은 테넌트 스코프(brandId) 일치도 검증해야. B2B는 수평이 더 흔함.
🔥 Q1-9. API 게이트웨이를 단일 RBAC 관문으로 둔 이유와 우회/SPOF 리스크는? (실제 코드)
왜: 중앙 집행은 깔끔하나 SPOF·우회가 단골 반론.
핵심: 권한 해석을 게이트웨이로 단일화(다운스트림 JWT 재파싱 제거). 우회 방어 = X-Gateway-Header 신뢰 마커로 경유 여부 판별, 미경유 시 RbacHeaderInjectingFilter fallback 인가. SPOF는 게이트웨이 다중화로 대응.
🔥 Q1-10. 클라이언트가 X-Function-Permissions 권한 헤더를 위조해서 보내면? (실제 코드)
왜: 헤더 기반 권한 전파의 핵심 공격 벡터. 못 막으면 RBAC 전체 무력화.
핵심: 게이트웨이가 클라 권한 헤더를 권위 값으로 덮어쓴다 — 외부 주입 X-*-Permissions는 진입 시 폐기되고 토큰 해석 결과로 재설정. 다운스트림은 X-Gateway-Header 마커 있는 요청만 신뢰.
🔥 Q1-11. 객체레벨 인가가 목록 API(유저 100건)에서 N+1 인가 쿼리 유발 안 하나? (2차 심화)
왜: 단건은 OK지만 목록은 행 단위로 "대상 역할 조회" N번. permissionByRole는 역할→권한만 캐시, 유저→역할은 못 흡수.
핵심: 단건(상세·수정)은 1건이라 문제없음. 목록은 권한 조건을 WHERE로 pushdown(권한 있는 행만 조회) or 유저→역할도 캐시. "함수레벨로 1차 거르고 객체레벨은 단건에만". 흡수 못 함을 먼저 인정이 핵심.
🔥 Q1-12. 토큰 RBAC 캐시 10분 — "즉시 무효화(0초)"라는데 SHA-256 키라 역추적 되나? (논리 모순)
왜: Result "즉시 0초" ↔ 토큰 캐시 10분 충돌. 면접관이 정확히 찌름.
핵심: permissionByRole(1분)은 @CacheEvict로 즉시. 단 토큰 캐시(10분, SHA-256 키)는 역추적 불가 → "유저→토큰해시 역인덱스(Redis Set)"로 함께 evict해야 "0초"가 참. 없으면 최대 10분 stale → JTI 블랙리스트 교차 체크로 보강. "캐시는 성능, 블랙리스트는 보안 — 보안 우선." (역인덱스 유무 본인 확인)
Q1-13. 응답 PII 마스킹을 진입점 인가가 아닌 AOP(@MaskIfNoPermission)로 한 이유? (실제 코드)
왜: "인가는 진입점에서 끝"이라는 통념을 넘는지.
핵심: 진입점은 "API 호출 가능?"만 — 응답 본문 필드(전화번호)는 통과. 권한 없는 운영자에게도 목록은 보이되 PII만 010-****-5678로 가리려면 응답 직렬화 시점 DtoMaskingAspect 필드 마스킹. 인가를 다층(진입점+응답)으로.
Q1-16. (트렌드) 토큰 탈취 후 다른 클라이언트 재사용 방어는? DPoP 고려했나?
왜: RFC 9700이 sender-constraining(mTLS/DPoP) 권고. bearer 한계.
핵심: 현재 bearer면 "DPoP 미적용, bearer+RTR+JTI 블랙리스트 조합으로 방어" 명시적 포지셔닝. DPoP(proof-of-possession) 설명 가능하면 가점.
2. 관측성 (OpenTelemetry / SigNoz)
🔥 Q2-1. OTel Agent와 @WithSpan SDK 동시 사용 시 GlobalOpenTelemetry 중복 초기화는?
왜: 이력서에 "주의사항"으로 적음 → 실제 겪었는지 검증.
핵심: Agent가 부트스트랩에서 이미 초기화 → SDK 재초기화 시 IllegalStateException. 정답은 SDK 초기화 코드를 아예 안 쓰는 것. @WithSpan/tracer는 Agent가 만든 인스턴스 자동 사용.
🔥 Q2-2. P95 알람 임계값을 어떻게 정했나? SLO와 어떻게 연결되나?
왜: 임의 숫자 vs SLO 역산은 SRE 성숙도의 결정적 차이. (SLO 부재 = Critical)
핵심: SLI(P95) → SLO(P95<1000ms, 30일 99%) → 알람은 SLO 위반 전 조기경고(burn rate). 안 했으면 솔직히 + "에러버짓 기반으로 발전시키고 싶다". RED 메서드 언급하면 깊이.
🔥 Q2-3. OTel Collector가 다운되면 애플리케이션에 무슨 일이? 어떻게 방어했나?
왜: 관측성이 서비스 critical path에 있으면 안 됨.
핵심: OTLP exporter 비동기 queue+retry. BatchSpanProcessor maxQueueSize·exporterTimeout으로 블로킹 방지. OTEL_EXPORTER_OTLP_TIMEOUT 짧게(5초). telemetry 유실되나 서비스 영향 없음 = "관측성 다운 ≠ 서비스 다운"의 실제 구현.
Q2-4. 커스텀 span attribute에 userId/orderId 같은 고카디널리티 값 썼나? ClickHouse 문제는?
왜: 카디널리티 폭증 = ClickHouse 디스크/쿼리 성능 최대 위협.
핵심: 고카디널리티는 span event/로그에. attribute는 enum 수준(order.status)으로 제한. Collector filter processor로 인제스트 전 제거.
Q2-5. 샘플링 전략 — head-based vs tail-based? 무엇을 썼나?
왜: 이력서 "샘플링=[실제 구성]" 미완성.
핵심: SigNoz 기본=head-based 100%. head 한계(에러/slow trace 놓침). tail은 Collector 버퍼링·메모리 부담. 소규모는 100% 수집+TTL 관리 합리적. 트래픽 증가 시 tail 전환 경로.
🔥 Q2-8. 로그 grep만 하다가 SigNoz로 실제 해결한 장애 사례 하나?
왜: 도구 도입보다 도구로 문제 해결한 경험. 수치 없는 이력서에서 최다 파고듦.
핵심: 구체 내러티브 — "카카오 OAuth 콜백 간헐 5초 지연 → trace에서 JWKS 외부호출 span 4.8초 + MDC 로그로 cache miss 직후 재조회 확인 → TTL 조정." flamegraph에서 뭘 보고 뭘 결론냈는지.
3. 도메인 설계 + DB 성능
🔥 Q3-1. N+1이 실제 발생한 걸 어떻게 진단했나?
왜: 코드 리뷰 추측 vs APM/로그 실측의 수준 차이.
핵심: SigNoz 트레이스에서 단일 요청 DB span 수 비정상(목록 20건에 쿼리 21개) + show_sql + EXPLAIN ANALYZE로 인덱스 미사용 확인.
🔥 Q3-2. Fetch Join과 @BatchSize 중 언제 뭘 쓰나? STAR 3엔 뭘 적용했나?
왜: 상호 대체 아님. 차이 모르는 경우 많음.
핵심: Fetch Join=단건 상세, 페이지네이션 없을 때. @BatchSize=목록(Pageable)/컬렉션 클 때. 목록에 Pageable 있으면 Fetch Join 단독은 HHH90003004 경고 → @BatchSize/DTO projection.
🔥 Q3-3. Fetch Join에 Pageable 붙이면 무슨 일이?
왜: 이 함정을 아는지로 즉각 수준 판별.
핵심: Hibernate가 SQL LIMIT 못 쓰고 전체 결과를 메모리에 올린 뒤 애플리케이션에서 자름(HHH90003004) → OOM. 해결: 목록 @BatchSize/DTO+별도 count, 상세 Fetch Join.
🔥 Q3-4. 복합 인덱스 컬럼 순서를 어떻게 정했나? 저카디널리티 컬럼을 넣은 이유는?
왜: 컬럼 순서가 플래너의 인덱스 사용 결정 핵심.
핵심: 등치(=) 필터 앞, 범위(BETWEEN) 뒤. 저카디널리티라도 항상 WHERE 포함+고카디널리티와 결합하면 선두 가능(예: (franchisor_id, content_type, created_at DESC)). MySQL은 선두 컬럼 매칭 원칙 절대적.
Q3-6. 3종 컨텐츠(Notice/Agreement/Document)를 어떻게 테이블 모델링했나?
왜: 공통 라이프사이클 다형 엔티티 모델링 = JPA 실무 검증 전형.
핵심: STI(단일테이블)=null 많으나 조인 없어 조회 빠름(목록 중심이면 합리). 조인테이블=정규화 좋으나 조인 비용. "어떤 걸 골랐고 왜 쿼리 패턴에 맞는지".
4. 결제·빌링
🔥 Q4-1. (memberId, billingDate) 유니크 제약 — 두 스레드 동시 INSERT면?
왜: 유니크 제약은 race를 막는 게 아니라 한쪽에 에러를 줌. 예외처리가 멱등성 완성.
핵심: 위반 시 DataIntegrityViolationException/DuplicateKeyException → catch → "이미 처리됨" skip. 또는 INSERT ... ON CONFLICT DO NOTHING. (이력서에 이 한 줄 추가 필요)
🔥 Q4-2. 결제 일배치가 PG 성공 후 DB 저장 실패(partial failure)면 멱등성은?
왜: DB 유니크는 이 케이스 못 막음. 인지 여부 테스트.
핵심: 토스 빌링 API에 Idempotency-Key 헤더로 (memberId+billingDate) 키 전달 → 동일 키 재호출 시 PG가 중복 차단+이전 결과 반환. DB 유니크 + PG idempotency 이중 방어.
🔥 Q4-3. 멀티 인스턴스에서 일배치가 두 인스턴스 동시 실행되면?
왜: Spring Scheduler 단독 = 인스턴스마다 실행. DB 유니크만으론 동시 PG 호출 못 막음.
핵심: 방어1=DB 유니크. 방어2=토스 Idempotency-Key. 근본=ShedLock @SchedulerLock(lockAtMostFor=deadlock 방지, lockAtLeastFor=조기 재실행 방지). "현재 단일이면 인지하고 ShedLock 도입 고려" 정직하게.
Q4-4. 유니크 제약 vs 비관적 락 — 결제 배치에 왜 유니크 제약을?
왜: 중복 방지 여러 패턴 중 선택 근거.
핵심: 비관적 락=락 대기·커넥션 hold·deadlock 위험. 유니크 제약=DB 원자 보장, retry-safe, 단방향 INSERT 적합. 일배치는 충돌 빈도 낮고 재실행 안전성 중요 → 유니크 제약.
Q4-6. 토스 Webhook(결제 이벤트) 위조 방지는?
왜: webhook 위조 = 실제 청구 없이 멤버십 활성화. 결제 핵심 공격벡터.
핵심: 토스 제공 서명(HMAC-SHA256)으로 바디 검증, 불일치 즉시 거부 + 토스 IP 화이트리스트. 미적용이면 "개선 계획 있다" 정직하게.
Q4-8. 무료체험 중 빌링키 등록 후 종료 직전 삭제로 결제 회피 가능한가?
왜: 비즈니스 로직 취약점(OWASP Business Logic).
핵심: 자동전환 시점 빌링키 존재 사전검증, 없으면 결제 없이 cancelled. "종료 N일 전 결제수단 등록 필수 알림 + 없으면 자동 해지" 정책으로 UX+보안 동시.
🔥 Q4-9. 추천인 리워드 멱등 지급을 어떻게? (실코드 정정)
왜: 돈이 걸린 "정확히 1회"를 분산락 없이.
핵심(정정): (memberId, rewardDate) 유니크가 아니라 조건부 UPDATE ... SET GRANTED WHERE status=PENDING의 row count로 선점 → 같은 트랜잭션에서 발급. 동시 배치 2개면 진 쪽 row 0 → skip. 외부 분산락·outbox 불필요. PK = (membershipId, promotionId), 1딜=1리워드.
Q4-10. 리워드 D+8 기준과 환불 처리는? (실코드)
왜: 결제·리워드 엣지 케이스를 정책 수준에서.
핵심: D+8 = 첫 결제 + 8일(전액환불 가능기간 종료) + 1h. D+8 이내 환불이면 배치 지급 제외(MIN(paidAt) 기준). 지급 후 환수(클로백) 미구현 = 백로그로 인지. "버그 아니라 범위 결정"으로 답.
🔥 Q4-11. 추천코드 셀프추천·가짜 피추천인 자가파밍(Sybil)은? (실코드)
왜: 그로스 루프 = 어뷰징 루프.
핵심: 자기추천 차단(userId), 추천 쿨다운 999개월(생애1회), 키=(사업자번호,종사업자번호,plan) NULL-safe. B2B 사업자번호 유니크가 구조적 방어. ⚠️ 잔존: 다른 사업자번호 shell 자가소비+D+8후 환불 → fraud 휴리스틱+클로백. 블랭킷 캡은 정상 파워추천인 훼손이라 정답 아님.
Q4-12. 환불 자체의 멱등성(이중 환불)은? (논리 비대칭)
왜: 결제는 유니크로 막았는데 환불은? 더블클릭=금전 손실.
핵심: 환불도 토스 Idempotency-Key + DB측 paymentKey당 누적환불액 ≤ 결제액 불변식을 비관락 검증. 부분환불 history가 단일 진실 원천. "결제 멱등성과 대칭"이 정답.
Q4-13. 정액 할인이 정가를 초과하면? (실코드)
왜: 결제 금액 경계 처리 디테일.
핵심: 음수 방지 0원 클램프 — 1원 아님(PG가 1원 결제를 거부하므로 0원). 무료 처리 흐름과 정합.
5. 아키텍처 공통 (시니어 판별)
Q5-1. "비동기 재시도 큐"가 알림톡·ERP·Deauth 3곳에 등장 — 같은 구현체? 별도?
왜: 횡단 관심사 공통화 설계 능력. 시니어는 반복 패턴 인식.
핵심: 별도면 "중복 인식했으나 범위 달라 개별 구현, 공통 모듈화는 다음 단계." 공통화면 "RetryQueue 도메인 서비스+어댑터별 주입." "얼마나 했나"보다 "반복을 봤을 때 반응".
Q5-2. 상태 전이 guard 로직은 서비스 if문인가 도메인 객체 내부인가?
왜: 서비스 if = 빈혈 도메인 모델. 시니어가 차이 인식하는지.
핵심: enum transitionTo(next)/allowedTransitions()로 도메인 레이어 차단이 이상적. 서비스 if면 "전이 단순해 선택, 복잡해지면 Spring State Machine 고려" 근거.
Q5-3. 멤버십 해지예약 상태에서 권한(canOrder) 회수 시점은? (STAR1↔STAR4 연결)
왜: 상태머신과 권한 모델 결합 지점을 설계한 사람만 정확히 답함.
핵심: 해지예약=이미 결제한 기간이라 권한 유지, 실제 cancelled 전환 시 회수. 회수=JTI 블랙리스트(즉시) vs 만료(1h) 중 무엇·왜. "연체→자동해지 시 즉시 회수=JTI 블랙리스트"와 연결.
Q5-4. 트래픽 10배면 이 아키텍처에서 먼저 병목 되는 곳은?
왜: 현재 규모엔 맞다 인정하면서 한계를 아는지(종합 질문).
핵심: SigNoz EC2 단독→ClickHouse 쓰기·디스크. Redis 단일→ElastiCache 클러스터/Read Replica. Scheduler 배치→처리시간이 다음 배치 전 미완료(Spring Batch 청크/파티셔닝). 확장 경로 명확히.
📌 모의면접 추천 순서: 🔥 표시부터. 특히 Q2-2(SLO), Q3-1~3(N+1), Q4-1~3(멱등성/멀티인스턴스)은 답변을 글로 써보고 입으로 1번 말해볼 것. "모르면 정직하게 + 향후 방향 제시"가 시니어 면접의 안전한 패턴.
