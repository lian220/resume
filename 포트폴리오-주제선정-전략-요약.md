## 핵심 전제 — "왜 뻔한 포트폴리오가 안 통하나"
- **바이브 코딩 시대**: AI로 웬만한 CRUD는 누구나 뚝딱 만듦 → 단순 CRUD·차별화 없는 프로젝트는 경쟁력 없음.
- **평가 방식이 바뀜**: 코드 구현력 < "진짜 이해했나 / 주도적으로 설계했나 / 결과를 검증할 수 있나". 라이브 코딩·설계 토론·코드 리뷰 전형으로 이동.
- **역할 분담 (이게 전부)**:
  - 🤖 **AI에게 맡길 것**: 코드 구현, 패턴/알고리즘, 테스트 코드, 디버깅
  - 🧑 **내가 주도할 것**: 문제 정의 · 설계 · 품질 관점 **트레이드오프 판단** · 구체적 **수치 측정** · 결과 **검증**
- AI 활용으로 코드 구현이 쉬워졌으니, 한 단계 높은 주제에 도전하고 **설계·품질·수치**로 차별화하라.

## AI 시대 포트폴리오 4대 방향성
1. **도메인 관심·관찰력** — 왜 이 문제를 다루는가 (관심·관찰·의도)
2. **문제 설정·해결 능력** — 문제를 구체적으로 정의 → 설계 → AI 구현 → 검증
3. **공학적 접근** — 설계 패턴·품질 지표를 주도적으로 측정 (AI 코드를 그냥 쓰지 않음)
4. **면접관이 직접 확인 가능한 결과물** — 배포 URL · GitHub 소스 · README(아키텍처·**트레이드오프 기록**) · 가능하면 프로젝트 설명 유튜브

## 🎯 6가지 포트폴리오 주제 유형

### ① 일상의 문제에서 출발
내가 직접 경험·관찰한 문제 = 가장 설득력 있는 동기. 면접관도 즉시 이해.
- 선착순 예매/대기열 → 자원 경쟁, 분산 락, 대기열 큐, 용량 계획, 부하 테스트
- 데이팅앱/추천 서비스 → 위치 기반 추천, 지리 인덱싱, 추천 정밀도
- 포트폴리오 자동 평가 시스템 (구현 난이도·AI 생성 코드·보안·쿼리 최적화·확장성 평가)
- ※ 주제가 평범하거나 완벽 구현이 아니어도, **공학적 접근 + 주도적 설계 + 결과 해석**이 있으면 좋은 포트폴리오.

### ② AI 코딩 환경 자체를 포트폴리오로
AI 코딩의 문제점을 직접 발견·개선한 경험을 정리.
- AI 문제점: 할루시네이션, 보안 취약점, 성능 무지, **회귀 버그**, 컨텍스트 윈도우 한계, 의존성 문제
- "CRUD를 만들며 이런 문제를 직접 찾아 공학적 원리로 개선했고, AI 잘 쓰는 기법을 정리했다" → 차별화된 포트폴리오

### ③ 빅테크 기술발표/장애사례(포스트모템) 재현
기술 블로그·포스트모템·세션에서 접근 방식·설계 의도만 가져와 **내 관점으로 직접 구현·검증**.
- 예: 트위터 타임라인 캐시 전략, 우버 실시간 위치 추적, 네이버페이 주문 메시지 무중단 전환
- 국내외 빅테크·세미나 자료 활용

### ④ 오픈소스 기여 또는 코드베이스 분석
실무 규모 코드를 다뤄본 경험 어필. **거창할 필요 없음.**
- 내가 실제 쓰는 작은 라이브러리의 이슈 찾기 → 코드 분석 → 원인 파악 → 개선 시도 → PR
- **PR이 거절·미머지여도 OK** — 시도한 과정·기록 자체가 포트폴리오 (코드 이해력 + 실무 규모 문제 발견 + 커뮤니티 참여 의지)

### ⑤ 소규모라도 실제 사용자가 있는 프로젝트
유저 10명이라도 있으면 완전히 다른 포트폴리오.
- 수치가 가상이 아닌 **실측값** + 실제 피드백 + 실제 배포 사이클 = "운영해 봤다"는 차별화
- 친구/소규모 커뮤니티/오픈카톡에 배포 → 피드백 수집 → 개선 → 재배포. **규모보다 사이클**이 중요.

### ⑥ 부트캠프 프로젝트 (차별화 필수)
부트캠프 프로젝트도 훌륭하지만 면접관은 "찍어낸 것"을 알아봄.
- 면접관이 식상해하는 신호: 동일 기관·동일 주제·유사 디자인 / 팀 프로젝트인데 본인 기여 불명확 / 결과물만 있고 why·공학적 원리·수치 없음
- 차별화 방법: **본인 기여 명확화** + 기술 선택의 **근거·트레이드오프 기록** + 아키텍처·품질 등 공학적 원리 + 위 ①~⑤ 관점을 접목

## 📌 추가 강조 — 지속적 기록 (포트폴리오보다 강력할 수 있음)
- **블로그 · GitHub · 유튜브**에 개발 이력·고민·기술·분석을 꾸준히 기록.
- 포트폴리오 = 취업용 단발성 결과물 / 블로그·깃헙 = 지속적 성격의 기록물 → 평소 관심사·태도·기술 수준·관리 습관을 보여줌.
- ⚠️ **역효과 주의**: 단순 복붙·얕은 글·일상 블로그 / 오래 방치된 블로그(차라리 안 보여주기) / 외부 활동·수상에 과도하게 치중("회사 일보다 외부 활동" 오해)

## 📊 실제 유저가 없어도 성능 수치 만드는 법
- 공학적 기준으로 목표 설정: **Google SRE식 SLO**, 응답시간 **P95 < N ms** 등 객관적 지표 + 업계 레퍼런스 참조
- **부하 생성기(k6 등)**로 가상 사용자 생성 → 부하 테스트 → 응답/처리량 기록 → 아키텍처 개선하며 향상
- 표현은 공학적 품질 지표로: "3초 걸리더라"(❌) → **P95 / TPS / Latency**(✅)

## ✅ 제출 전 셀프 체크리스트
- [ ] 내가 왜 이 주제를 선택했는지 설명할 수 있는가?
- [ ] 직접 설정한 공학적 원리·품질 기준·측정 수치가 있는가?
- [ ] 배포 URL, GitHub 소스 코드 URL이 있는가?
- [ ] 아키텍처를 정리하고 프로젝트를 설명하는 영상이 있는가?
- [ ] AI를 어떻게 활용했고 어떻게 검증했는지 설명할 수 있는가?
- [ ] 핵심 코드 설계를 직접 구두로 설명할 수 있는가?
- [ ] 기술 선택 시 어떤 트레이드오프를 고려했는지 말할 수 있는가?
- [ ] 다른 지원자와 확연히 구분되는 나만의 관점이 있는가?
> 체크가 많을수록 잘 준비된 포트폴리오.

## 한 줄 결론
> 구현만 하고 끝나는 단순 CRUD는 차별화 없음. **문제 발견 → 명확한 설계 + 트레이드오프 기록 → AI 주도 활용 + 비판적 검토 → 공학적 수치로 개선·검증**. 모두 할 필요 없이 **나에게 맞는 몇 가지를 골라 어필 요소로**.
